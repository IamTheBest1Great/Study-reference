## 23. Situation-Based / Scenario Questions

> These are the most important section for senior interviews. Interviewers describe a real-world problem and ask you to diagnose, explain, or solve it.

---

### 🔴 Transactions & Database Scenarios

**S1.** Your `@Transactional` method is NOT rolling back even though a `RuntimeException` is thrown. What are the possible causes?
> — Exception is caught and swallowed internally. Method is called from within the same class (proxy bypass). Method is `private`. Exception is checked and `rollbackFor` is not set. The method isn't on a Spring-managed bean.

**S2.** You have two services — `ServiceA.methodA()` calls `ServiceB.methodB()`. Both are `@Transactional`. `methodA` throws an exception after `methodB` completes. Does `methodB` roll back?
> — Depends on propagation. With default `REQUIRED`, both run in the same transaction — yes, both roll back. With `REQUIRES_NEW` on `methodB`, it already committed — no rollback for `methodB`.

**S3.** You're seeing `LazyInitializationException` in production when returning an entity from a REST controller. What are the causes and how do you fix it?
> — Hibernate session closed before lazy association accessed. Fixes: use DTO projection, `JOIN FETCH`, `@EntityGraph`, `@Transactional` on service, or disable OSIV (not recommended to enable it).

**S4.** Your database is running N+1 queries for a single API call — 1 query for users + N queries for each user's orders. How do you diagnose and fix it?
> — Enable `spring.jpa.show-sql=true`, `logging.level.org.hibernate.SQL=DEBUG`. Fix: use `JOIN FETCH` in JPQL, `@EntityGraph`, `@BatchSize`, or a DTO projection with a single join query.

**S5.** After deploying, you discover that `ddl-auto=update` silently dropped a column in production. What happened and how do you prevent this?
> — Hibernate auto DDL is destructive and unpredictable. Switch to `ddl-auto=validate` or `none` in production and use Flyway/Liquibase for schema migrations.

**S6.** Two instances of your Spring Boot service are both processing the same scheduled job concurrently, causing duplicate processing. How do you fix it?
> — Use ShedLock library with a distributed lock backed by DB/Redis. Or use Quartz cluster mode with JDBC job store. Alternatively, Kafka/queue-based triggering with single consumer group.

**S7.** A transaction is deadlocking intermittently in production. How do you investigate?
> — Enable DB-level deadlock logging. Ensure consistent lock acquisition order across transactions. Shorten transaction scope. Switch from `SERIALIZABLE` to `REPEATABLE_READ`. Consider optimistic locking with `@Version`.

**S8.** `@Transactional` with `REQUIRES_NEW` is not creating a new transaction in your test. What is likely happening?
> — `@Transactional` in tests rolls back by default. If the same thread is used, nested `REQUIRES_NEW` behaves differently in tests vs production. Use `@Rollback(false)` or test in a real DB.

### 🔴 Spring AOP & Proxy Scenarios

**S9.** You have a `@Cacheable` method in `ServiceA` that calls `ServiceAImpl.someOtherMethod()` from within the same class. Caching isn't working. Why?
> — Spring AOP proxy is bypassed on self-invocations. The call goes to `this`, not the proxy. Fix: inject self-reference using `@Autowired ServiceA self`, use `@Lookup`, or restructure to separate class.

**S10.** Your `@Async` method is executing synchronously. What are the common causes?
> — `@EnableAsync` is missing. Method is called from within the same class. Method is `private`. Exception in the async executor is swallowed. Executor misconfiguration.

**S11.** An `@Around` advice on your service is not executing. What could be wrong?
> — Aspect class not detected by Spring (missing `@Component` or not in scan path). `@EnableAspectJAutoProxy` missing (Spring Boot auto-configures it but verify). Pointcut expression is incorrect. CGLIB proxy not enabled and target class doesn't implement interface.

**S12.** You want to log execution time for all methods in `com.example.service` package. Write the `@Around` aspect and explain why it's better than putting `System.currentTimeMillis()` in every method.
> — Implement `@Around("execution(* com.example.service.*.*(..))")` with `ProceedingJoinPoint`, capture time before/after `proceed()`. Centralizes cross-cutting concern, no code duplication, can be toggled off.

### 🔴 Security Scenarios

**S13.** Your JWT tokens are being accepted after a user logs out. How do you implement token revocation?
> — Store revoked JWTs in Redis with TTL equal to token expiry. On each request, check if token ID (`jti`) is in revocation store before validating claims. Alternatively, use short-lived access tokens + long-lived refresh tokens and revoke only refresh tokens.

**S14.** An internal service-to-service call is receiving a 401 Unauthorized. What do you check?
> — Verify the `Authorization` header is propagated between services. Check Feign interceptor or `WebClient` filter that adds the JWT. Verify the JWT signature key matches between the auth server and resource server. Check `SecurityFilterChain` for the internal service.

**S15.** A user with role `ADMIN` is getting 403 Forbidden on an endpoint annotated `@PreAuthorize("hasRole('ADMIN')")`. What are possible causes?
> — Role stored as `ADMIN` but Spring expects `ROLE_ADMIN` prefix (use `hasAuthority('ROLE_ADMIN')` or `hasRole('ADMIN')` consistently). Method security not enabled (`@EnableMethodSecurity` missing). `SecurityContextHolder` not populated correctly.

**S16.** Your Spring Boot API is vulnerable to CSRF attacks. But you already disabled CSRF protection because it's a REST API. Is that safe? When is it safe to disable?
> — Safe for stateless REST APIs that use JWT (not session/cookie auth). Not safe if you use session-based auth with cookies. Rule: if your API uses `Authorization: Bearer` header, disabling CSRF is safe. If it uses `HttpOnly` session cookies, CSRF protection must stay on.

### 🔴 Performance & Scaling Scenarios

**S17.** Your Spring Boot application is slow after running for a few hours but recovers after a restart. What could be the cause?
> — Likely a memory leak causing GC pressure (increasing full GC pauses). Take heap dump. Check for caches growing unboundedly, connection leaks, `ThreadLocal` not cleaned, event listeners not deregistered, or a growing in-memory queue.

**S18.** Your application throws `HikariPool-1 - Connection is not available, request timed out after 30000ms`. What does this mean and how do you fix it?
> — Connection pool exhausted. Connections not returned (not closed, long transactions). Fix: increase `maximumPoolSize` (but first investigate root cause), shorten transaction scope, ensure connections are closed, monitor `hikari.active` metric, check for slow queries holding connections.

**S19.** You deployed a Spring Boot service and it takes 15 seconds to start. How do you reduce startup time?
> — Enable lazy initialization (`spring.main.lazy-initialization=true`). Restrict component scan. Remove unused starters. Use Spring Boot 3.2+ AOT/native. Use Spring Boot layered JARs with JVM CDS. Profile startup with `-Dspring.startup.trace=true` or `ApplicationStartup` metrics.

**S20.** Your application has a slow endpoint — `GET /products` is taking 3 seconds. How do you systematically diagnose it?
> — Enable SQL logging to find N+1 or missing indexes. Check if `COLLSCAN` / slow query in DB logs. Add Actuator `/metrics` for endpoint timing. Check if external API call is blocking. Check thread pool saturation. Add `@Timed` or manual timing. Use async profiler on a live instance.

**S21.** You need to process 1 million records from a CSV file uploaded via a Spring Boot endpoint. The current implementation loads everything into memory and crashes with OOM. How do you redesign it?
> — Stream the `MultipartFile` directly. Use `BufferedReader` over input stream. Process in batches (e.g., 500 records at a time) with a dedicated `@Async` executor. Use Spring Batch with chunk-oriented processing. Store the file in S3, enqueue a message in Kafka/RabbitMQ, process asynchronously.

**S22.** Your API Gateway (Spring Cloud Gateway) is not forwarding requests to a downstream service correctly after scaling the downstream service to 3 instances. What do you check?
> — Ensure `@LoadBalanced` is on `WebClient` or `RestTemplate`. Verify service is registered in Eureka/Consul with correct instance metadata. Check load balancing strategy. Verify health check passes so unhealthy instances are excluded.

### 🔴 Configuration & Deployment Scenarios

**S23.** Your Spring Boot application reads the correct config locally but not in production (Docker). Why?
> — Environment variables override `application.properties` but may not match property names. Check case sensitivity (`SPRING_DATASOURCE_URL` vs `spring.datasource.url`). Verify `SPRING_PROFILES_ACTIVE` is set. Check if `application-prod.yml` is included in the JAR. Check if config server is unreachable.

**S24.** You have a Spring Boot service with `@Value("${api.key}")` that fails to start with `Could not resolve placeholder 'api.key'`. How do you fix it?
> — Add `api.key=` to `application.properties`. Or use `@Value("${api.key:defaultValue}")` for a default. Or use `@ConfigurationProperties` with required=false. Verify the property is in the active profile's file.

**S25.** You need to support multiple databases in one Spring Boot app — a read replica for queries and primary for writes. How do you implement it?
> — Configure two `DataSource` beans. Use `@Primary` on the write DataSource. Use `AbstractRoutingDataSource` to route based on `@Transactional(readOnly = true)` flag via `TransactionSynchronizationManager.isCurrentTransactionReadOnly()`. Configure separate `EntityManagerFactory` for each DataSource.

**S26.** After deploying a new version, requests are failing with 503 because the old pod shuts down too fast. How do you implement graceful shutdown?
> — Set `server.shutdown=graceful`. Configure `spring.lifecycle.timeout-per-shutdown-phase=30s`. Ensure Kubernetes `preStop` hook and `terminationGracePeriodSeconds` align. Spring Boot will wait for active requests to complete before shutting down the Tomcat thread pool.

### 🔴 Messaging & Async Scenarios

**S27.** Your `@KafkaListener` is consuming the same message multiple times. How do you ensure exactly-once processing?
> — Set `enable-auto-commit=false`, use manual `AckMode.MANUAL`. Implement idempotency using a processed message ID store (Redis/DB). Enable Kafka transactions with `KafkaTransactionManager`. Use `isolation.level=read_committed` on the consumer.

**S28.** Messages in your RabbitMQ dead letter queue are growing. What does this mean and how do you handle it?
> — Messages failed processing (exception thrown) and were rejected after retries. Investigate exceptions in consumer. Implement proper retry with backoff using `ListenerContainerFactory`. Add `DeadLetterExchange` handler to reprocess or alert.

**S29.** You need to ensure a downstream REST call is made only after a database transaction commits successfully. How do you implement this?
> — Use `@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)`. Publish a Spring event inside the transaction. The listener fires only after successful commit. Alternatively, use the outbox pattern — write to an outbox table in the same transaction, a separate process reads and sends.

### 🔴 Testing Scenarios

**S30.** Your `@WebMvcTest` is failing because a `@Component` bean used by the controller cannot be found. What is the issue?
> — `@WebMvcTest` only loads the web layer (controllers, filters, `WebMvcConfigurer`). It doesn't load `@Service` or `@Repository` beans. Add `@MockBean` for missing dependencies, or use `@SpringBootTest` if full context is needed.

**S31.** You want to write an integration test for a repository that writes to PostgreSQL and then reads back. How do you do it without affecting production data?
> — Use Testcontainers with `@Testcontainers` + `@Container` to start a real PostgreSQL. Annotate test with `@Transactional` for auto-rollback, or use `@Sql` for setup/teardown scripts. Or use a dedicated test schema.

**S32.** Your tests are slow because every test class loads a new Spring context. How do you fix it?
> — Ensure all test classes use the same context configuration (same `@SpringBootTest` setup, same mocks). Spring caches contexts by configuration key. Avoid `@DirtiesContext` unless necessary. Use `@MockBean` carefully — adding new `@MockBean` creates a new context.

### 🔴 Architecture & Design Scenarios

**S33.** You need to add logging of all incoming HTTP requests and outgoing responses (method, URL, status, duration) without modifying every controller. How?
> — Implement a `HandlerInterceptor` with `preHandle` and `afterCompletion`. Or use a `CommonsRequestLoggingFilter` (Spring built-in). Or write an `@Around` aspect on all controller methods. Or configure `LoggingFilter` with `WebMvcConfigurer`.

**S34.** You are building a payment service where the same payment request might be sent twice (network retry). How do you ensure idempotency in Spring Boot?
> — Accept an `Idempotency-Key` header. Store (key → response) in Redis with TTL. On each request, check if key exists. If yes, return cached response. If no, process and store atomically. Mark DB rows with idempotency key.

**S35.** Your Spring Boot monolith needs to be split into microservices. How would you approach it incrementally?
> — Use Strangler Fig pattern. Identify bounded contexts. Extract one service at a time. Introduce API gateway. Add service discovery (Eureka). Implement distributed tracing from day one. Keep shared DB temporarily, then migrate to per-service DB. Use event-driven communication where possible.

**S36.** You need to build a feature that sends email notifications after order placement, but the current implementation makes the user wait while email is sent. How do you fix it?
> — Decouple notification from request using `@Async` service or Kafka/RabbitMQ event. Controller returns response immediately after order saved. A separate consumer picks up the `OrderPlacedEvent` and sends email asynchronously.

**S37.** You need to implement a global request timeout — any request taking longer than 5 seconds should return 503. How do you implement this in Spring Boot?
> — Implement a `Filter` or `HandlerInterceptor` with timeout. Or use Spring Cloud Gateway's `RequestRateLimiter` / `CircuitBreaker` with timeout filter. Or configure Resilience4j `@TimeLimiter` on service methods. Or configure `server.servlet.session.timeout`.

**S38.** Your Spring Boot service's memory usage keeps growing in a Kubernetes pod until it gets OOMKilled. Walk through your investigation.
> — Verify `-XX:+UseContainerSupport` and `-XX:MaxRAMPercentage=75` are set. Take Actuator `/heapdump`. Analyze with Eclipse MAT for object retention. Check for: unbounded caches, static collections, `ThreadLocal` leaks, Hibernate session leaks, large incoming payloads. Monitor `jvm.memory.used` metric trend.

**S39.** Two Spring Boot services communicate via REST. How do you propagate the correlation/trace ID across service boundaries?
> — Use Micrometer Tracing (Spring Boot 3) which automatically propagates `traceid` and `spanid` via HTTP headers (`B3` or `W3C TraceContext` format). Configure Zipkin/Jaeger as backend. On the receiving service, the tracing auto-config picks up the incoming trace context. Optionally use `MDC.put("traceId", ...)` for log correlation.

**S40.** You need to run different logic based on an environment — stub implementation in dev, real implementation in prod. How do you implement it cleanly in Spring Boot?
> — Define an interface. Create two implementations annotated with `@Profile("dev")` and `@Profile("prod")`. Spring injects the correct one based on active profile. Alternatively, use `@ConditionalOnProperty`.
