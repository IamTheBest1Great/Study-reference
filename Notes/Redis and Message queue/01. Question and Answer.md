### Cache Invalidation

- What is cache invalidation? Why is it hard?
- What are the strategies for cache invalidation?
  - TTL-based expiry, event-driven invalidation (pub/sub), write-through, version-based keys
- What is a stale cache? What is the impact?
- What is a **cache stampede** (thundering herd)? How does it occur?
- How do you prevent cache stampede?
  - `SET key value EX ttl NX` (mutex lock), probabilistic early expiry, background refresh, `cache.sync=true` (Spring)
- What is **cache penetration**? How do you prevent it?

- What is **hotspot key**? How do you handle it?
  - One key gets hammered (e.g., viral post). Fix: local in-process cache, key splitting, read replicas

## 9. Redis — Situation-Based Questions

**S1.** Your Redis cache hit rate is dropping significantly. How do you investigate?
> — Check `INFO stats`: `keyspace_hits` vs `keyspace_misses`. Inspect TTL settings — keys expiring too fast. Check if keys are being evicted (`evicted_keys` counter). Verify cache-aside logic is correctly setting keys after DB reads. Check for cache avalanche (mass expiry). Review key naming for correctness.

**S2.** Your Redis `maxmemory` is full and you're seeing OOM errors. The eviction policy is `noeviction`. What do you do?
> — Immediately: switch eviction policy to `allkeys-lru` or `volatile-lru` using `CONFIG SET maxmemory-policy allkeys-lru`. Long-term: increase `maxmemory`, scale horizontally with Redis Cluster, audit big keys (`--bigkeys`), reduce TTLs, review data no longer needed. Never use `noeviction` for pure cache use cases.

**S3.** Two instances of your service are both trying to acquire a Redis distributed lock using `SETNX`. Occasionally, both get the lock simultaneously. What is the bug?
> — Using two commands `SETNX key; EXPIRE key ttl` is not atomic — process can die between them, leaving key without expiry (permanent lock). Fix: use single atomic command `SET key uuid EX ttl NX`. Also ensure UUID is used as value so only the owner can release.

**S4.** Your Redis latency is spiking every 60 seconds. What is the likely cause?
> — `BGSAVE` (RDB snapshot) running every 60 seconds — the `fork()` system call causes a brief pause. Or `BGREWRITEAOF` triggering. Also check `latency monitor` and `slowlog`. If large dataset, `fork()` COW causes memory pressure. Consider disabling RDB snapshots if not needed, or tuning save intervals.

**S5.** You're using `KEYS *` in production to find all keys matching a pattern, but it's freezing Redis for 2 seconds. How do you fix it?
> — `KEYS` is O(N) and blocks the event loop. Replace with `SCAN 0 MATCH pattern COUNT 100` in a loop. `SCAN` is non-blocking (cursor-based) and doesn't block other clients. In production, also rename or disable `KEYS` command via `rename-command KEYS ""`.

**S6.** After a Redis Sentinel failover, some of your application instances are still writing to the old primary (now a replica), causing data loss. How do you fix it?
> — Applications must use the Sentinel-aware client that auto-discovers the new primary after failover. Do not hardcode the Redis IP — connect via Sentinel endpoints. Enable `min-replicas-to-write 1` on the old primary to refuse writes when it becomes a replica. Use client-side retry on `READONLY` errors.

**S7.** Your Redis Cluster is rejecting multi-key operations like `MSET` with a `CROSSSLOT` error. How do you fix it?
> — Multi-key operations in Redis Cluster require all keys to hash to the same slot. Use hash tags `{}` to force co-location: `MSET {user:1}.name "Alice" {user:1}.age "30"` — both keys hash on `user:1`. Redesign data model to keep related keys in same slot.

**S8.** You implemented a cache-aside pattern, but users are seeing stale data for up to 5 minutes after an update. Why?
> — TTL is set to 5 minutes and cache is not invalidated on writes. Fix: on every update/delete in DB, also call `DEL key` or update the cache with new value (`SET key newValue EX ttl`). Consider write-through pattern. Or reduce TTL. Or use event-driven invalidation via pub/sub.

**S9.** Your Redis memory is 80% full, and you see `expired_keys` growing rapidly. What is happening and is it a problem?
> — Many keys are expiring. This is normal behavior of cache. Growing `expired_keys` just means the lazy + active expiry is working. Concern: if `maxmemory` is being hit and evictions are happening (`evicted_keys > 0`), that's where to focus. Monitor `used_memory` trend and `mem_fragmentation_ratio`.

**S10.** You need to implement a leaderboard for a gaming app that shows top 100 players by score, updated in real time. How do you design it with Redis?
> — Use Sorted Set: `ZADD leaderboard <score> <userId>` on each score update. `ZREVRANGE leaderboard 0 99 WITHSCORES` returns top 100. `ZINCRBY leaderboard <delta> <userId>` for score increments. `ZRANK leaderboard <userId>` for a player's rank. `ZCARD` for total player count.

## 15. RabbitMQ — Situation-Based Questions

**S1.** Messages are accumulating in a RabbitMQ queue (queue depth growing) and consumers are not processing them. How do you investigate?
> — Check consumer count (`list_queues name messages consumers`). If 0 consumers: consumer application is down — restart it. If consumers present: check for consumer exceptions causing channel close without requeue. Check `prefetch_count` — if set to 1 and consumers are slow, it throttles delivery. Check for unacked messages piling up (`messages_unacknowledged`). Increase consumer instances or prefetch count.

**S2.** A message gets rejected by your consumer repeatedly and keeps cycling back to the queue, causing an infinite loop. How do you fix it?
> — This is a "poison message". Fix: configure DLX on the queue. On `nack`, use `requeue=false` — message goes to DLX → DLQ. Add max retry logic using `x-death` count in message headers. If retry count exceeds limit, route to permanent DLQ for inspection. Never `nack` with `requeue=true` on persistent failures.

**S3.** You're losing messages after a RabbitMQ broker restart. What configuration changes do you need?
> — Three things must be durable: (1) Exchange must be declared `durable=true`. (2) Queue must be declared `durable=true`. (3) Messages must be published with `deliveryMode=2` (persistent). All three must be set — durable queue with transient message still loses messages on restart.

**S4.** A RabbitMQ queue has grown to 10 million messages and consumers are backed up. How do you recover without data loss?
> — Add more consumer instances (horizontal scaling) — they auto compete for messages. Increase `prefetch_count`. If queue is overflow, consider enabling lazy queue mode (`x-queue-mode=lazy`) to spill to disk. If you must shed load, consider `x-max-length` with `drop-head` policy for new queues. Use `rabbitmqadmin` or management API to monitor progress.

**S5.** You need to implement a delayed job queue in RabbitMQ — a task that should execute after 30 seconds. How do you design it?
> — Option 1: Use `rabbitmq_delayed_message_exchange` plugin — set `x-delay: 30000` in message headers. Option 2: Dead letter approach — publish to a "holding" queue with `x-message-ttl=30000` and no consumers, DLX set to the actual work exchange. After 30s, message expires → routes to work queue via DLX.

**S6.** Two microservices need to communicate where Service A publishes an event and multiple Service B instances should each receive a copy. How do you design this?
> — Use a Fanout exchange or Topic exchange. Each Service B instance declares its own exclusive queue and binds to the exchange. This way each instance gets its own copy. If you want only one instance of Service B to process (competing consumers), all instances bind the same queue to the exchange.

**S7.** During a RabbitMQ cluster network partition, some nodes become unreachable. Messages are being published to a partition minority node. What configuration prevents data loss?
> — Set `pause-minority` partition handling — minority side pauses all operations. Use Quorum Queues which require majority acknowledgement (Raft-based). Enable publisher confirms so producers know if messages were accepted. Set `min-masters` policy on quorum queues.

**S8.** Your RabbitMQ consumer is processing messages very slowly because it's doing heavy computation synchronously. How do you improve throughput?
> — Increase `prefetch_count` (currently might be 1 — limiting to one message at a time). Add more consumer processes/threads. Offload heavy computation to an async thread pool within consumer, ack immediately after placing on internal queue. Scale consumer service horizontally — multiple instances auto-compete on the same queue.

**S1.** A Kafka consumer group is lagging by 10 million messages and the lag keeps growing. What do you do?
> — Diagnose with `kafka-consumer-groups.sh --describe` — find which partitions lag. Check consumer throughput — is it processing too slowly? Increase consumer instances (up to number of partitions). Increase `max.poll.records`. Check if `max.poll.interval.ms` is too low — if processing takes too long, consumer gets kicked out, causing rebalances and lag. Profile consumer code for bottlenecks. If lag is permanent, consider adding partitions (with caution).

**S2.** You're seeing duplicate messages being consumed after a Kafka broker restart. How do you prevent duplicates?
> — Producer side: enable `enable.idempotence=true` with `acks=all`. This prevents duplicate on retry. Consumer side: use `commitSync()` after processing (not auto-commit) to avoid consuming same message on restart. Implement idempotent processing — check processed message IDs in DB/Redis before processing. For end-to-end exactly-once: use Kafka Transactions with `isolation.level=read_committed`.

**S3.** One Kafka partition is receiving 90% of all messages while other partitions are idle. What causes this and how do you fix it?
> — Hot partition due to key skew — many messages share the same key, all routing to same partition. Fix: audit key selection — choose a high-cardinality key that distributes evenly. Use a composite key. For keyless messages, switch to round-robin or sticky partitioner. Consider salting the key (append random suffix) if ordering within that key isn't strictly needed.

**S4.** Your Kafka topic has 3 partitions but your consumer group has 6 consumers. What happens?
> — Only 3 consumers receive messages (one per partition). The other 3 are idle. Kafka assigns at most one consumer per partition per group. To utilize all 6 consumers, increase partitions to 6 or more. You can increase partition count, but cannot decrease without deleting and recreating the topic.

**S5.** After increasing partitions on a Kafka topic, some consumers in your group are consuming messages out of order. Why?
> — Message ordering in Kafka is guaranteed only within a single partition, not across partitions. When you increase partitions, keys that previously mapped to partition 1 might now map to partition 3 (hash mod new-partition-count changes). Consumers reading from different partitions see messages in different orders. For strict ordering on a key, never increase partitions (or use a side-channel for ordering).

**S6.** Your Kafka producer is throwing `NotEnoughReplicasException` intermittently. What is the cause?
> — `acks=all` is set and `min.insync.replicas` is not being met — some ISR replicas are lagging or offline. Check `UnderReplicatedPartitions` metric. Fix: bring lagging brokers back online. Temporarily lower `min.insync.replicas` to `1` for recovery (data risk). Long-term: fix network/disk issues on replica brokers. Increase `replica.lag.time.max.ms` if replicas are temporarily slow.

**S7.** You need to replay all Kafka messages from the last 7 days for a new microservice that just went live. How do you do it?
> — Create a new consumer group for the new service. Use `--reset-offsets --to-datetime "2025-01-01T00:00:00.000"` with `kafka-consumer-groups.sh` before starting consumption. Or use `seek()` in the consumer code. Make sure topic retention is at least 7 days (`log.retention.hours=168`). The new consumer group starts from the reset offset without affecting other consumer groups.

**S8.** A Kafka consumer is failing with `org.apache.kafka.clients.consumer.CommitFailedException: Commit cannot be completed since the group has already rebalanced`. What happened?
> — Consumer took too long to process messages — exceeded `max.poll.interval.ms`. Kafka removed it from the group and triggered rebalance. The consumer tried to commit but the group no longer owns those partitions. Fix: reduce processing time per batch, increase `max.poll.interval.ms`, decrease `max.poll.records` to process smaller batches, or move slow processing to async thread (ack after processing completes).

**S9.** You need to implement an event-driven saga pattern across 3 microservices using Kafka. Service A places order → Service B reserves inventory → Service C processes payment. How do you handle the compensation if payment fails?
> — Each step publishes success/failure event. Design compensating transactions: if payment fails, publish `payment.failed` event. Service B listens and publishes `inventory.release` event. Service A listens and marks order as failed. Use a dedicated saga orchestrator (Service D) that tracks state and sends commands. Use Kafka Streams or a state machine. Ensure idempotency at each step using message ID deduplication.

**S10.** Your Kafka cluster has 3 brokers. One broker disk is full and it's an ISR member for many partitions. What do you do?
> — Immediately: `kafka-reassign-partitions.sh` to move some partitions off the full broker. Enable `log.dirs` on a second disk path on that broker if available. Delete old log segments if retention allows. Long-term: add a new broker and rebalance. Use Cruise Control for automated rebalancing. Set `log.retention.bytes` per topic to prevent runaway growth. Alert on disk usage > 70% in future.

**S11.** You want to implement exactly-once message processing in Kafka end-to-end. What must be configured?
> — Producer: `enable.idempotence=true`, `acks=all`, `transactional.id=unique-id`. Use `beginTransaction/commitTransaction`. Consumer: `isolation.level=read_committed`. Consumer commit: wrap DB write + offset commit in same Kafka transaction (if sink is Kafka). If sink is a DB (non-Kafka), use outbox pattern + idempotent DB upsert with message ID as deduplication key.

## 25. Redis vs Kafka vs RabbitMQ Comparisons

### Core Philosophy

| Aspect | Redis | RabbitMQ | Kafka |
|---|---|---|---|
| Primary Role | In-memory data store / cache | Message broker | Distributed event log |
| Protocol | Redis protocol (RESP) | AMQP (+ MQTT, STOMP) | Binary TCP (proprietary) |
| Message Model | Ephemeral (Pub/Sub) / Streams | Push to consumers | Pull by consumers |
| Message Persistence | Optional (Streams: yes; Pub/Sub: no) | Optional (durable queues) | Always (log-based) |
| Message Retention | Until consumed / TTL | Until consumed (deleted after ack) | Configurable (days/weeks) |
| Message Replay | Streams only | No | Yes |
| Throughput | Very high (in-memory) | Moderate (~50K msg/s) | Very high (~millions/s) |
| Ordering | Per list/stream | Per queue | Per partition |
| Consumer Groups | Streams only | Competing consumers (one group) | Named groups, multiple groups |
| Latency | Sub-millisecond | Low (1–10ms) | Low (single-digit ms) |

### When to Use Which

- **Use Redis when**: You need caching, session storage, real-time leaderboards, rate limiting, distributed locks, pub/sub for simple notifications, and data fits in memory.
- **Use RabbitMQ when**: You need flexible routing (exchanges, bindings, routing keys), request-reply patterns, task queues with priorities, complex routing logic, short-lived messages, protocol flexibility (AMQP/MQTT/STOMP).
- **Use Kafka when**: You need event streaming at massive scale, message replay, audit logs, event sourcing, multiple independent consumer groups, long-term retention, high throughput, CDC (Change Data Capture).

| At-most-once | Auto-ack | autoAck=true | auto.commit with no retry |
| At-least-once | Streams + XACK | manual ack + requeue | manual commit + retry |
