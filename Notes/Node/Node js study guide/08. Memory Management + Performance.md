# Node.js Memory Management & Performance – Complete Guide

Memory management in Node.js is a critical aspect of building scalable, production‑grade applications. Unlike languages with manual memory management (C/C++) or automatic reference counting, Node.js relies on **V8's generational garbage collector**. Understanding how memory is allocated, how garbage collection works, and how to profile and optimize memory usage is essential for preventing leaks, reducing latency spikes, and maximizing throughput.

---

## 1. V8 Memory Structure & Heap Space Tuning

### Memory Layout of a Node.js Process

A Node.js process's memory is divided into several regions:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Node.js Process Memory                             │
├─────────────────────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────────────────────────┐│
│  │                              Resident Set (RSS)                          ││
│  │  ┌───────────────────┐  ┌───────────────────┐  ┌───────────────────┐    ││
│  │  │   V8 Heap         │  │   Native Memory   │  │   Code & Stack    │    ││
│  │  │  (JS Objects)     │  │  (Buffers, C++)   │  │  (Executable)     │    ││
│  │  └───────────────────┘  └───────────────────┘  └───────────────────┘    ││
│  └─────────────────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────────────────┘
```

- **V8 Heap**: Stores JavaScript objects, strings, closures, arrays. Managed by V8's garbage collector.
- **Native Memory**: Memory allocated outside V8 (e.g., `Buffer`, C++ addons, libuv handles).
- **Code & Stack**: Compiled machine code and call stacks.

### V8 Heap Spaces (Detailed)

V8 divides the managed heap into **spaces** with different purposes:

| Space | Purpose | Typical Size | GC Frequency |
|-------|---------|--------------|--------------|
| **New Space** | Newly allocated objects (young generation) | 1‑16 MB | Very frequent (minor GC) |
| **Old Space** | Objects surviving multiple minor GCs | Up to heap limit | Less frequent (major GC) |
| **Large Object Space** | Objects > 256KB (bypassed New Space) | Variable | Major GC only |
| **Code Space** | Compiled JIT code | ~10‑50 MB | Rarely collected |
| **Map Space** | Hidden classes (shapes) | ~5‑20 MB | Rarely collected |
| **Read‑Only Space** | Immutable data (strings, numbers) | ~1‑5 MB | Never collected |

**Large Object Space Optimization:**
Objects larger than the threshold (`--max-semi-space-size / 2`) are allocated directly in **Large Object Space**. This prevents expensive copying during minor GC.

### Tuning Heap Size Limits

```bash
# Set maximum Old Space size (default: ~1.4 GB on 64-bit, ~512 MB on 32-bit)
node --max-old-space-size=4096 server.js   # 4 GB

# Set New Space (semi‑space) size (default: platform dependent, ~16‑32 MB)
node --max-semi-space-size=128 server.js   # 128 MB (per semi‑space)

# Set initial Old Space size
node --initial-old-space-size=1024 server.js

# Memory reduction (aggressive GC) for memory‑constrained environments
node --optimize-for-size server.js
```

**Choosing the Right Values:**
- **Increase `--max-old-space-size`** for memory‑intensive applications (e.g., large in‑memory caches, image processing)
- **Increase `--max-semi-space-size`** for applications that create many short‑lived objects (e.g., web servers with high request rate) to reduce promotion frequency
- **Use `--optimize-for-size`** in containers with strict memory limits (e.g., 256 MB) to favor GC over performance

### Monitoring Memory Usage

## 2. Garbage Collection Cycles in Depth

### Generational Hypothesis

V8's GC strategy is based on the **generational hypothesis**: most objects die young. By segregating objects by age, the GC can focus on collecting young objects frequently (minor GC) and only occasionally collect the entire heap (major GC).

### Minor GC (Scavenge)

**Algorithm:** Cheney's copying collector (semispace copying)

**Process:**
1. New Space is divided into two equal **semi‑spaces**: **From‑space** (active) and **To‑space** (inactive).
2. All allocations happen in From‑space.
3. When From‑space fills up, minor GC is triggered.
4. Live objects are **copied** to To‑space, compacting them.
5. Dead objects are simply abandoned (no sweeping needed).
6. The semi‑spaces swap roles.
7. Objects that have survived **two minor GCs** are **promoted** to Old Space.

**Characteristics:**
- **Extremely fast** (only touches live objects)
- **Pause time proportional to live set**, not heap size
- **Memory overhead:** 50% of New Space is always unused (To‑space)

### Major GC (Mark‑Sweep‑Compact)

**Triggered when:**
- Old Space reaches a certain threshold
- Incremental marking completes
- Failed allocation in Old Space

**Phases:**

1. **Mark (Concurrent/Incremental)**: Traces object graph from roots, marking reachable objects. V8 does this **concurrently** in background threads and **incrementally** interleaved with JavaScript execution to minimize pause times.

2. **Sweep (Concurrent)**: Frees memory of unmarked objects. Runs in background threads.

3. **Compact (Stop‑the‑World)**: Moves surviving objects together to eliminate fragmentation. Only performed when fragmentation is high.

**Pause Times:** Major GC pauses are typically **10‑50 ms** but can reach hundreds of milliseconds for large heaps with heavy fragmentation.

### Understanding GC Cycles with Tracing Flags

```bash

node --trace-gc server.js

# Output:

# [4116:0x5a00000]     8321 ms: Mark-sweep 8.5 (12.0) -> 6.2 (16.0) MB, 5.2 / 0.0 ms

# Verbose GC tracing

# Trace GC and write to file for analysis
node --trace-gc --trace-gc-verbose server.js 2>&1 | tee gc.log
```

## 4. Flame Graphs — Visualizing Performance

A **flame graph** is a visualization of stack traces collected during profiling. Each box represents a function call; width indicates time spent.

### Generating Flame Graphs with 0x

```bash
# Install
npm install -g 0x

# Generate flame graph for your app
0x -- node server.js

# With delay (capture after warmup)
0x --delay 5 -- node server.js

# Output opens automatically in browser
```

**Reading a Flame Graph:**
- **X‑axis**: Alphabetical stack ordering (not time)
- **Y‑axis**: Stack depth (callers below, callees above)
- **Width**: CPU time spent in that function (wider = more time)
- **Colors**: Random or semantic (e.g., red for native code, green for JS)

**What to Look For:**
- **Wide plateaus**: Functions consuming significant CPU
- **Tall narrow towers**: Deep recursion (possible stack overflow risk)
- **Repeated patterns**: Opportunity for memoization

### Generating Flame Graphs with Linux perf

```bash

perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg
```

## 5. Detecting and Fixing Memory Leaks

### Common Causes of Memory Leaks in Node.js

| Leak Type | Example | Detection |
|-----------|---------|-----------|
| **Global variables** | `global.cache = {}` without cleanup | Heap snapshot shows growing object |
| **Forgotten timers** | `setInterval()` not cleared | Idle heap grows over time |
| **Event listeners** | `emitter.on()` without `off()` | Retained emitter and closure |
| **Closures capturing large objects** | Function retains reference to large array | Heap snapshot shows unexpected references |
| **Streams not destroyed** | `fs.createReadStream()` without `destroy()` | Native memory leak (external) |
| **Promises never resolved/rejected** | Pending promises hold references | Heap snapshot shows `Promise` objects |
| **Caches without TTL** | In‑memory `Map` growing unbounded | Heap grows linearly |

### Detecting Leaks with Heap Snapshots

**Step‑by‑Step Process:**

1. **Capture baseline snapshot** after server startup
2. **Simulate load** (e.g., run 10,000 requests)
3. **Force GC** (`global.gc()`) then capture second snapshot
4. **Compare snapshots** in Chrome DevTools

```javascript
// Programmatic heap snapshot for automated leak detection
const v8 = require('node:v8');
const fs = require('node:fs');

function captureSnapshot(label) {
  const snapshot = v8.writeHeapSnapshot(`./snapshots/${label}-${Date.now()}.heapsnapshot`);
  console.log(`Snapshot saved: ${snapshot}`);
}

// In your test flow
captureSnapshot('before');
await runLoadTest();
global.gc();
captureSnapshot('after');
```

**Analyzing the Comparison:**
- Sort by **Retained Size** descending
- Look for objects that exist in "after" but not in "before"
- Check the **Retainers** path to understand why they're held

### Using `llnode` for Post‑Mortem Analysis

```bash
# Generate core dump on crash
ulimit -c unlimited
node --abort-on-uncaught-exception server.js

# Analyze core dump with llnode
llnode node -c core.12345

v8 findjsinstances MyClass
```

### Real‑World: Fixing a Common Event Emitter Leak

```javascript
// ❌ Leaky code
class ConnectionPool {
  constructor() {
    this.connections = [];
  }
  
  addConnection(conn) {
    this.connections.push(conn);
    conn.on('data', (data) => {
      this.handleData(data);  // this is captured in closure
    });
    // Missing: cleanup when connection closes
  }
}

// ✅ Fixed
class ConnectionPool {
  constructor() {
    this.connections = new Set();
  }
  
  addConnection(conn) {
    this.connections.add(conn);
    
    const dataHandler = (data) => this.handleData(data);
    conn.on('data', dataHandler);
    
    conn.once('close', () => {
      conn.removeListener('data', dataHandler);
      this.connections.delete(conn);
    });
  }
}
```

### Real‑World: Unbounded Cache with TTL

```javascript
// ❌ Memory leak: unbounded cache
const cache = new Map();

function getCached(key) {
  return cache.get(key);
}
function setCached(key, value) {

  allowStale: false
});
```

## 6. Buffer Optimization

### Understanding Buffer Memory

`Buffer` objects represent **raw binary data** and are allocated **outside the V8 heap** (native memory). They are **not subject to V8's garbage collection** in the same way — they use a separate reference‑counting mechanism.

```javascript
// Creating Buffers
const buf1 = Buffer.alloc(1024);          // Zero‑filled, safe
const buf2 = Buffer.allocUnsafe(1024);    // Faster, may contain old data
const buf3 = Buffer.from('hello');        // From string/array
```

### Buffer Pool (Internal Optimization)

Node.js maintains an internal **Buffer pool** (8 KB by default) to avoid frequent allocation/deallocation of small Buffers.

```javascript
// These small allocations use the pool
Buffer.alloc(1024);   // Uses pool
Buffer.alloc(4096);   // Uses pool
Buffer.alloc(8192);   // Bypasses pool (slow path)
```

### Avoiding Buffer Memory Leaks

```javascript
// ❌ Leak: Buffer never freed because it's referenced in closure
function readFileLeaky(path) {
  const buffer = Buffer.alloc(10 * 1024 * 1024); // 10 MB
  fs.readFile(path, (err, data) => {
    data.copy(buffer);
    processBuffer(buffer);
    // buffer still referenced by closure until callback completes
  });
}

// ✅ Better: Release reference ASAP
function readFile(path) {
  fs.readFile(path, (err, data) => {
    processBuffer(data);  // Use directly, no extra allocation
  });
}
```

### Optimizing Buffer Operations

**Zero‑Copy Transfers with Transferable Objects:**

```javascript
const { parentPort } = require('node:worker_threads');

const buffer = Buffer.alloc(1024 * 1024 * 100); // 100 MB

// Transfer ownership (zero copy, buffer becomes unusable here)
parentPort.postMessage(buffer, [buffer.buffer]);
```

**Using SharedArrayBuffer:**

```javascript
// Main thread
const sharedBuffer = new SharedArrayBuffer(1024 * 1024 * 10); // 10 MB
const worker = new Worker('./worker.js', { workerData: { sharedBuffer } });

// Worker thread
const { workerData } = require('node:worker_threads');
const sharedArray = new Uint8Array(workerData.sharedBuffer);
// Both threads see the same memory — no copying!
```

**Buffer Slicing (O(1), No Copy):**

```javascript
const largeBuffer = fs.readFileSync('huge-file.bin');
const firstKB = largeBuffer.subarray(0, 1024);  // Creates view, not copy
// firstKB shares underlying memory with largeBuffer
```

### Monitoring External Memory

```javascript
// Track Buffer memory separately from V8 heap

  const mem = process.memoryUsage();
  console.log(`External memory: ${(mem.external / 1024 / 1024).toFixed(2)} MB`);
  console.log(`ArrayBuffers: ${(mem.arrayBuffers / 1024 / 1024).toFixed(2)} MB`);

// It's NOT managed by V8 GC
```

## 7. C++ Bindings with Node‑API (N-API)

Node‑API (formerly N‑API) is a **stable ABI** (Application Binary Interface) for building native addons. It isolates addons from changes in the underlying V8 engine, ensuring compatibility across Node.js versions.

### Why Node‑API?

- **Stable across Node.js versions** — no recompilation needed for minor upgrades
- **Managed memory** — V8 GC can track native resources
- **Thread‑safe** — supports async work and worker threads
- **Simpler than raw V8/NAN** — less boilerplate, more maintainable

### Basic Node‑API Addon Example

```cpp
// addon.cc
#include <node_api.h>
#include <cstring>

// Synchronous function: add two numbers
napi_value Add(napi_env env, napi_callback_info info) {
  napi_status status;
  size_t argc = 2;
  napi_value args[2];
  
  status = napi_get_cb_info(env, info, &argc, args, nullptr, nullptr);
  if (status != napi_ok) return nullptr;
  
  double a, b;
  status = napi_get_value_double(env, args[0], &a);
  if (status != napi_ok) return nullptr;
  status = napi_get_value_double(env, args[1], &b);
  if (status != napi_ok) return nullptr;
  
  napi_value result;
  status = napi_create_double(env, a + b, &result);
  if (status != napi_ok) return nullptr;
  
  return result;
}

// Initialize module
napi_value Init(napi_env env, napi_value exports) {
  napi_status status;
  napi_value fn;
  
  status = napi_create_function(env, nullptr, 0, Add, nullptr, &fn);
  if (status != napi_ok) return nullptr;
  
  status = napi_set_named_property(env, exports, "add", fn);
  if (status != napi_ok) return nullptr;
  
  return exports;
}

NAPI_MODULE(NODE_GYP_MODULE_NAME, Init)
```

**Build configuration (`binding.gyp`):**
```json
{
  "targets": [
    {
      "target_name": "addon",
      "sources": [ "addon.cc" ]
    }
  ]
}
```

**Usage in JavaScript:**
```javascript
const addon = require('./build/Release/addon');
console.log(addon.add(3, 5)); // 8
```

### Async Work with Node‑API (Thread Pool)

```cpp
// async-addon.cc
#include <node_api.h>
#include <thread>
#include <chrono>

struct AsyncWorkData {
  napi_async_work work;
  napi_deferred deferred;
  napi_ref callback_ref;
  double input;
  double result;
};

void ExecuteWork(napi_env env, void* data) {
  AsyncWorkData* workData = static_cast<AsyncWorkData*>(data);
  
  // Simulate CPU‑intensive work (runs in thread pool)
  std::this_thread::sleep_for(std::chrono::milliseconds(100));
  workData->result = workData->input * workData->input;
}

void CompleteWork(napi_env env, napi_status status, void* data) {
  AsyncWorkData* workData = static_cast<AsyncWorkData*>(data);
  
  napi_value result;
  napi_create_double(env, workData->result, &result);
  
  // Resolve the Promise
  napi_resolve_deferred(env, workData->deferred, result);
  
  // Cleanup
  napi_delete_async_work(env, workData->work);
  delete workData;
}

napi_value ComputeSquareAsync(napi_env env, napi_callback_info info) {
  napi_status status;
  size_t argc = 1;
  napi_value args[1];
  
  status = napi_get_cb_info(env, info, &argc, args, nullptr, nullptr);
  if (status != napi_ok) return nullptr;
  
  double input;
  status = napi_get_value_double(env, args[0], &input);
  if (status != napi_ok) return nullptr;
  
  // Create Promise
  napi_value promise;
  napi_deferred deferred;
  status = napi_create_promise(env, &deferred, &promise);
  if (status != napi_ok) return nullptr;
  
  // Setup async work
  AsyncWorkData* workData = new AsyncWorkData{
    .deferred = deferred,
    .input = input,
    .result = 0
  };
  
  napi_value resource_name;
  napi_create_string_utf8(env, "ComputeSquare", NAPI_AUTO_LENGTH, &resource_name);
  
  status = napi_create_async_work(
    env, nullptr, resource_name,
    ExecuteWork, CompleteWork,
    workData, &workData->work
  );
  if (status != napi_ok) {
    delete workData;
    return nullptr;
  }
  
  status = napi_queue_async_work(env, workData->work);
  if (status != napi_ok) {
    delete workData;
    return nullptr;
  }
  
  return promise;
}
```

### Managing Native Resources with Object Wrap

```cpp
// Database connection wrapper
class DatabaseConnection : public node::ObjectWrap {
public:
  static void Init(napi_env env, napi_value exports) {
    napi_value ctor;
    napi_define_class(env, "DatabaseConnection", NAPI_AUTO_LENGTH,
      New, nullptr, 0, nullptr, &ctor);
    
    // Store constructor for later use
    napi_create_reference(env, ctor, 1, &constructor_ref_);
    napi_set_named_property(env, exports, "DatabaseConnection", ctor);
  }
  
private:
  static napi_value New(napi_env env, napi_callback_info info) {
    napi_value new_target;
    napi_get_new_target(env, info, &new_target);
    
    DatabaseConnection* obj = new DatabaseConnection();
    obj->real_connection_ = open_real_db_connection();
    
    napi_wrap(env, new_target, obj,
      [](napi_env env, void* data, void* hint) {
        DatabaseConnection* obj = static_cast<DatabaseConnection*>(data);
        close_real_db_connection(obj->real_connection_);
        delete obj;
      },
      nullptr, nullptr);
    
    return new_target;
  }
  
  void* real_connection_;
  static napi_ref constructor_ref_;
};
```

### Node‑API Best Practices

1. **Use `node-addon-api`** — C++ wrapper that provides a more idiomatic API
   ```bash
   npm install node-addon-api
   ```

2. **Handle exceptions properly** — Node‑API functions return `napi_status`; check after each call

3. **Manage native memory carefully** — Use `napi_wrap` with a finalizer to free native resources when the JavaScript object is garbage collected

4. **Avoid blocking the event loop** — Use `napi_create_async_work` for CPU‑intensive or blocking operations

5. **Use `Napi::TypedArray` for binary data** — More efficient than copying Buffers

```cpp
// Efficient Buffer handling with node-addon-api
Napi::Value ProcessBuffer(const Napi::CallbackInfo& info) {
  Napi::Env env = info.Env();
  
  // Get Buffer as uint8_t* without copying
  Napi::Buffer<uint8_t> buffer = info[0].As<Napi::Buffer<uint8_t>>();
  uint8_t* data = buffer.Data();
  size_t length = buffer.Length();
  
  // Process data in place
  for (size_t i = 0; i < length; i++) {
    data[i] = ~data[i];  // Invert bits
  }
  
  return buffer;  // Return same buffer (modified)
}
```

## 8. Complete Performance Tuning Checklist

| Area | Check | Action |
|------|-------|--------|
| **Heap** | `heapUsed` growing continuously | Look for memory leaks, unbounded caches |
| **Heap** | `heapTotal` near `heap_size_limit` | Increase `--max-old-space-size` |
| **GC** | Frequent major GC (>1/sec) | Increase heap limit or reduce allocation rate |
| **GC** | Long GC pauses (>50ms) | Use `--max-semi-space-size` to tune New Space |
| **External** | `external` memory growing | Check Buffers, C++ addons for leaks |
| **Thread Pool** | High CPU with low throughput | Increase `UV_THREADPOOL_SIZE` |
| **Event Loop** | `perf_hooks` shows lag >10ms | Profile for blocking operations |
| **CPU** | High CPU usage | Use Clinic flamegraph to find hot functions |

| **C++ Bindings** | Node‑API, `node-addon-api` | Stable ABI, managed native resources, async work |

**Key Takeaways:**
1. **Monitor memory continuously** — use `process.memoryUsage()` and APM tools
2. **GC is not free** — tune heap sizes to balance memory and pause times
3. **Profile before optimizing** — use flame graphs to identify actual bottlenecks
4. **Prefer Node‑API for native addons** — it's stable and simpler than NAN
5. **Buffer operations can leak native memory** — track `external` memory separately
6. **Use bounded caches with TTL** — prevent unbounded heap growth
7. **Test under production‑like load** — memory leaks often only appear under sustained traffic
