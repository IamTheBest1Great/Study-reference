# Logging, Monitoring & Observability – Node.js Guide

Observability is the ability to understand the internal state of a system from its external outputs. In distributed Node.js applications, effective logging, metrics, and tracing are essential for debugging, performance optimization, and maintaining reliability. This guide covers structured logging with Pino and Winston, application performance monitoring (APM) tools, distributed tracing with OpenTelemetry, and metrics collection.

---

## 1. Structured Logging

Structured logging produces machine‑readable logs (usually JSON) that can be easily ingested by log aggregation systems like ELK, Loki, or Datadog. Unlike traditional text logs, structured logs enable powerful querying and analysis.

### Why Structured Logging?

| Traditional Logging | Structured Logging |
|---------------------|-------------------|
| `User 123 logged in` | `{ "level": "info", "message": "User logged in", "userId": 123, "timestamp": "..." }` |
| Hard to query by userId | Easily filter `userId=123` |
| Inconsistent format | Standardized schema |
| Difficult to aggregate metrics | Extract metrics from log fields |

### Pino — High‑Performance Logger

**Pino** is the fastest Node.js logger, designed for production with minimal overhead.

```javascript
const pino = require('pino');

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level: (label) => ({ level: label }),
    bindings: (bindings) => ({ pid: bindings.pid, host: bindings.hostname }),
  },
  timestamp: pino.stdTimeFunctions.isoTime,
  redact: {
    paths: ['password', 'email', 'token', 'authorization'],
    censor: '[REDACTED]'
  },
  transport: process.env.NODE_ENV === 'development' ? {
    target: 'pino-pretty',
    options: { colorize: true, translateTime: 'SYS:standard' }
  } : undefined
});

// Add request‑scoped child logger
app.use((req, res, next) => {
  req.log = logger.child({
    requestId: req.headers['x-request-id'] || crypto.randomUUID(),
    method: req.method,
    url: req.url
  });
  next();
});

// Usage
app.get('/users/:id', async (req, res) => {
  req.log.info({ userId: req.params.id }, 'Fetching user');
  
  try {
    const user = await db.users.findOne({ id: req.params.id });
    req.log.info({ userId: req.params.id, found: !!user }, 'User fetched');
    res.json(user);
  } catch (err) {
    req.log.error({ err, userId: req.params.id }, 'Failed to fetch user');
    res.status(500).json({ error: 'Internal server error' });
  }
});
```

### Winston — Feature‑Rich Logger

**Winston** offers more features out of the box, including multiple transports, log rotation, and custom formats.

```javascript
const winston = require('winston');
const { combine, timestamp, json, errors, metadata } = winston.format;

const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: combine(
    errors({ stack: true }),
    timestamp(),
    metadata({ fillExcept: ['message', 'level', 'timestamp'] }),
    json()
  ),
  defaultMeta: { service: 'user-service' },
  transports: [
    new winston.transports.Console({
      format: process.env.NODE_ENV === 'development'
        ? combine(winston.format.colorize(), winston.format.simple())
        : json()
    }),
    new winston.transports.File({ filename: 'error.log', level: 'error' }),
    new winston.transports.File({ filename: 'combined.log' }),
    new winston.transports.DailyRotateFile({
      filename: 'application-%DATE%.log',
      datePattern: 'YYYY-MM-DD',
      maxSize: '20m',
      maxFiles: '14d'
    })
  ],
  exceptionHandlers: [
    new winston.transports.File({ filename: 'exceptions.log' })
  ],
  rejectionHandlers: [
    new winston.transports.File({ filename: 'rejections.log' })
  ]
});

// Express middleware
app.use((req, res, next) => {
  req.logger = logger.child({
    requestId: req.headers['x-request-id'] || uuid(),
    method: req.method,
    path: req.path
  });
  
  const start = Date.now();
  res.on('finish', () => {
    req.logger.info({
      statusCode: res.statusCode,
      duration: Date.now() - start,
      userAgent: req.headers['user-agent']
    }, 'Request completed');
  });
  
  next();
});
```

logger.add(new DatadogTransport({

### Logging Best Practices

1. **Log at appropriate levels**: `trace`, `debug`, `info`, `warn`, `error`, `fatal`
2. **Include correlation IDs** to trace requests across services
3. **Redact sensitive data** (passwords, tokens, PII)
4. **Log structured data** — avoid string interpolation for dynamic values
5. **Never log to console in production** — use a file or remote transport
6. **Sample debug logs** in high‑volume environments

## 2. APM Tools (Application Performance Monitoring)

APM tools provide deep insights into application performance, including transaction traces, error tracking, and resource utilization.

### Elastic APM

```javascript
const apm = require('elastic-apm-node').start({
  serviceName: 'user-service',
  serverUrl: process.env.ELASTIC_APM_SERVER_URL,
  secretToken: process.env.ELASTIC_APM_SECRET_TOKEN,
  environment: process.env.NODE_ENV,
  captureBody: 'off',
  logLevel: 'info',
  transactionSampleRate: 0.5, // Sample 50% of transactions
  captureErrorLogStackTraces: 'always'
});

// Express integration is automatic — spans created for routes
app.get('/users/:id', async (req, res) => {
  const span = apm.startSpan('fetchUserFromDB');
  try {
    const user = await db.users.findOne({ id: req.params.id });
    span?.setLabel('userFound', !!user);
    res.json(user);
  } finally {
    span?.end();
  }
});

// Custom transaction
apm.startTransaction('backgroundJob', 'worker');
try {
  await processBatch();
  apm.endTransaction('success');
} catch (err) {
  apm.captureError(err);
  apm.endTransaction('failure');
}
```

### Datadog APM (dd‑trace)

```javascript
const tracer = require('dd-trace').init({
  service: 'user-service',
  env: process.env.NODE_ENV,
  version: process.env.npm_package_version,
  logInjection: true, // Inject trace ID into logs
  runtimeMetrics: true,
  profiling: true,
  sampleRate: 0.1
});

// Automatic instrumentation for Express, PostgreSQL, Redis, etc.

// Custom span
app.get('/users/:id', (req, res) => {
  tracer.trace('fetchUser', { resource: 'UserService' }, async (span) => {
    span.setTag('user.id', req.params.id);
    const user = await db.users.findOne({ id: req.params.id });
    span.setTag('user.found', !!user);
    res.json(user);
  });
});
```

### New Relic APM

```javascript
// newrelic.js
'use strict';
exports.config = {
  app_name: ['user-service'],
  license_key: process.env.NEW_RELIC_LICENSE_KEY,
  logging: {
    level: 'info'
  },
  distributed_tracing: {
    enabled: true
  },
  transaction_tracer: {
    record_sql: 'obfuscated',
    explain_threshold: 500
  },
  error_collector: {
    ignore_status_codes: [401, 404]
  }
};

// Usage — automatic instrumentation, plus custom
const newrelic = require('newrelic');

app.get('/users/:id', async (req, res) => {
  return newrelic.startSegment('fetchUser', true, async () => {
    const user = await db.users.findOne({ id: req.params.id });
    newrelic.addCustomAttribute('userFound', !!user);
    return user;
  });
});
```

### APM Metrics to Monitor

| Metric | Description | Alert Threshold |
|--------|-------------|-----------------|
| **Request Rate** | Requests per second | Baseline deviation |
| **Error Rate** | Percentage of 5xx responses | > 1% |
| **Response Time (p95)** | 95th percentile latency | > 500ms |
| **Apdex Score** | User satisfaction index | < 0.7 |
| **CPU/Memory Usage** | Resource consumption | > 80% |
| **Database Query Time** | Slow queries | > 100ms |

## 3. Distributed Tracing with OpenTelemetry

OpenTelemetry (OTel) is a vendor‑neutral standard for collecting telemetry data (traces, metrics, logs). It enables end‑to‑end visibility across distributed systems.

### OpenTelemetry Architecture

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           OpenTelemetry Pipeline                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   ┌─────────────┐     ┌─────────────────┐     ┌─────────────────┐           │
│   │  Node.js    │────▶│  OTel SDK      │────▶│  OTel Collector │           │
│   │  Service    │     │ (Instrumentation)│     │   (Agent)       │           │
│   └─────────────┘     └─────────────────┘     └────────┬────────┘           │
│                                                         │                     │
│                                                         ▼                     │
│                                              ┌─────────────────────┐         │
│                                              │  Backend            │         │
│                                              │ (Jaeger/Zipkin/     │         │
│                                              │  Datadog/Elastic)   │         │
│                                              └─────────────────────┘         │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Setting Up OpenTelemetry in Node.js

```javascript
// tracing.js
const { NodeSDK } = require('@opentelemetry/sdk-node');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');
const { OTLPTraceExporter } = require('@opentelemetry/exporter-trace-otlp-http');
const { Resource } = require('@opentelemetry/resources');
const { SemanticResourceAttributes } = require('@opentelemetry/semantic-conventions');
const { W3CTraceContextPropagator } = require('@opentelemetry/core');
const { diag, DiagConsoleLogger, DiagLogLevel } = require('@opentelemetry/api');

// Enable diagnostic logging in development
if (process.env.NODE_ENV === 'development') {
  diag.setLogger(new DiagConsoleLogger(), DiagLogLevel.DEBUG);
}

const sdk = new NodeSDK({
  resource: new Resource({
    [SemanticResourceAttributes.SERVICE_NAME]: 'user-service',
    [SemanticResourceAttributes.SERVICE_VERSION]: process.env.npm_package_version,
    [SemanticResourceAttributes.DEPLOYMENT_ENVIRONMENT]: process.env.NODE_ENV
  }),
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT || 'http://localhost:4318/v1/traces'
  }),
  instrumentations: [getNodeAutoInstrumentations()],
  textMapPropagator: new W3CTraceContextPropagator()
});

sdk.start();

// Graceful shutdown
process.on('SIGTERM', async () => {
  await sdk.shutdown();
  process.exit(0);
});
```

**Run with:**
```bash
node --require ./tracing.js app.js
```

### Manual Instrumentation

```javascript
const { trace } = require('@opentelemetry/api');

const tracer = trace.getTracer('user-service');

app.get('/users/:id', async (req, res) => {
  const span = tracer.startSpan('getUserById');
  span.setAttribute('user.id', req.params.id);

  try {
    const user = await db.users.findOne({ id: req.params.id });
    span.setAttribute('user.found', !!user);
    span.setStatus({ code: trace.SpanStatusCode.OK });
    res.json(user);
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: trace.SpanStatusCode.ERROR, message: err.message });
    res.status(500).json({ error: err.message });
  } finally {
    span.end();
  }
});
```

### Propagating Trace Context

OpenTelemetry automatically injects trace context into HTTP headers (`traceparent`). For message queues, you must manually propagate:

```javascript
// Producer (publishing to Kafka)
const { propagation, trace } = require('@opentelemetry/api');

const span = tracer.startSpan('publishOrderEvent');
const context = trace.setSpan(trace.context.active(), span);

const headers = {};
propagation.inject(context, headers);

await producer.send({
  topic: 'orders',
  messages: [{
    value: JSON.stringify(order),
    headers: headers
  }]
});
span.end();

// Consumer
const { propagation, trace } = require('@opentelemetry/api');

consumer.run({
  eachMessage: async ({ message }) => {
    const parentContext = propagation.extract(trace.context.active(), message.headers);
    const span = tracer.startSpan('processOrderEvent', {}, parentContext);
    
    try {
      await processOrder(JSON.parse(message.value));
    } finally {
      span.end();
    }
  }
});
```

## 4. Metrics

Metrics are numerical measurements collected over time, used for monitoring system health, capacity planning, and alerting.

### Prometheus Metrics with `prom-client`

```javascript
const client = require('prom-client');

// Create a Registry
const register = new client.Registry();
client.collectDefaultMetrics({ register, prefix: 'user_service_' });

// Custom metrics
const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Duration of HTTP requests in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.01, 0.05, 0.1, 0.5, 1, 2, 5, 10]
});

const httpRequestsTotal = new client.Counter({
  name: 'http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'route', 'status_code']
});

const activeConnections = new client.Gauge({
  name: 'active_connections',
  help: 'Number of active connections'
});

const dbQueryDuration = new client.Summary({
  name: 'db_query_duration_seconds',
  help: 'Duration of database queries in seconds',
  labelNames: ['query_type', 'table'],
  percentiles: [0.5, 0.9, 0.95, 0.99]
});

register.registerMetric(httpRequestDuration);
register.registerMetric(httpRequestsTotal);
register.registerMetric(activeConnections);
register.registerMetric(dbQueryDuration);

// Middleware to record metrics
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer();
  res.on('finish', () => {
    const route = req.route?.path || req.path;
    httpRequestsTotal.inc({
      method: req.method,
      route,
      status_code: res.statusCode
    });
    end({ method: req.method, route, status_code: res.statusCode });
  });
  next();
});

// Expose metrics endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});
```

```javascript
const { MeterProvider } = require('@opentelemetry/sdk-metrics');
const { PrometheusExporter } = require('@opentelemetry/exporter-prometheus');

const exporter = new PrometheusExporter({ port: 9464 });
const meterProvider = new MeterProvider();
meterProvider.addMetricReader(exporter);

const meter = meterProvider.getMeter('user-service');

const requestCounter = meter.createCounter('http_requests_total', {
  description: 'Total HTTP requests'
});

const requestDuration = meter.createHistogram('http_request_duration', {
  description: 'HTTP request duration',
  unit: 'ms'
});

app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    requestCounter.add(1, { method: req.method, route: req.path, status: res.statusCode });
    requestDuration.record(duration, { method: req.method, route: req.path });
  });
  next();
});
```

## 5. Unified Observability Pipeline

A production‑grade observability setup integrates logs, metrics, and traces.

```javascript
// observability.js — unified setup
const pino = require('pino');
const { trace, context, propagation } = require('@opentelemetry/api');

// Configure logger with trace ID injection
const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    log(object) {
      const span = trace.getSpan(context.active());
      if (span) {
        const spanContext = span.spanContext();
        object.trace_id = spanContext.traceId;
        object.span_id = spanContext.spanId;
      }
      return object;
    }
  }
});

// Middleware to correlate everything
app.use((req, res, next) => {
  const requestId = req.headers['x-request-id'] || crypto.randomUUID();
  
  // Create child logger with request context
  req.log = logger.child({ requestId });
  
  // Start span for this request
  const span = tracer.startSpan(`${req.method} ${req.path}`, {
    attributes: {
      'http.method': req.method,
      'http.url': req.url,
      'http.request_id': requestId
    }
  });
  
  // Store span in context
  context.with(trace.setSpan(context.active(), span), () => {
    res.on('finish', () => {
      span.setAttribute('http.status_code', res.statusCode);
      span.end();
      
      // Log request completion with metrics
      req.log.info({
        statusCode: res.statusCode,
        duration: Date.now() - req._startTime,
        userAgent: req.headers['user-agent']
      }, 'Request completed');
    });
    
    req._startTime = Date.now();
    next();
  });
});
```

## Summary Table

| Pillar | Tool Options | Purpose |
|--------|--------------|---------|
| **Logging** | Pino (fast), Winston (feature‑rich) | Debug, audit, error investigation |
| **APM** | Elastic APM, Datadog, New Relic | Performance bottlenecks, error tracking |
| **Distributed Tracing** | OpenTelemetry + Jaeger/Zipkin | End‑to‑end request flow |
| **Metrics** | Prometheus + Grafana | Capacity planning, alerting |
| **Visualization** | Kibana (logs), Grafana (metrics), Jaeger UI (traces) | Dashboards and exploration |

**Key Takeaways:**
1. **Use structured logging** — JSON logs are essential for aggregation
2. **Inject trace IDs into logs** — correlate logs with traces for debugging
3. **Start with OpenTelemetry** — vendor‑neutral and future‑proof
4. **Collect RED metrics** — Rate, Errors, Duration for every service
5. **Alert on symptoms, not causes** — high error rate, not "database connection failed"
6. **Sample high‑volume traces** — avoid overwhelming observability backends
