# Node.js Module System Internals – Complete Guide

The module system is the backbone of Node.js code organization. Understanding how modules are resolved, loaded, cached, and executed is essential for debugging complex issues, optimizing startup performance, and building robust applications. This guide covers everything from the differences between CommonJS and ES Modules to advanced patterns like loader hooks and dependency injection.

---

## 1. ESM vs CommonJS

Node.js supports two module systems: **CommonJS** (the original) and **ECMAScript Modules** (ESM, the standard). They have fundamental differences in syntax, loading behavior, and interoperability.

### Comparison Table

| Feature | CommonJS | ECMAScript Modules (ESM) |
|---------|----------|--------------------------|
| **Syntax** | `require()`, `module.exports` | `import`, `export` |
| **Loading** | Synchronous, runtime | Asynchronous, static analysis |
| **File Extensions** | `.js`, `.cjs` | `.mjs`, `.js` (with `"type": "module"`) |
| **Top‑Level `await`** | Not allowed | Allowed |
| **`this` at top level** | `module.exports` | `undefined` |
| **Live Bindings** | No (copy of exports) | Yes (imported values update) |
| **Tree Shaking** | Not possible | Possible (static analysis) |
| **Circular Dependencies** | Partially works (exports object) | Works with live bindings |
| **`__dirname`, `__filename`** | Available | Not available (use `import.meta.url`) |
| **`require.resolve()`** | Available | Use `import.meta.resolve()` |

### Syntax Examples

**CommonJS:**
```javascript
// math.js
const add = (a, b) => a + b;
const PI = 3.14159;

module.exports = { add, PI };
// or: exports.add = add;

// main.js
const math = require('./math');
console.log(math.add(2, 3)); // 5
```

**ES Modules:**
```javascript
// math.js
export const add = (a, b) => a + b;
export const PI = 3.14159;
export default function multiply(a, b) { return a * b; }

// main.js
import multiply, { add, PI } from './math.js';
console.log(add(2, 3)); // 5
```

### Interoperability (Dual Packages)

A package can support both systems using conditional exports:

```json
{
  "name": "my-lib",
  "exports": {
    "import": "./dist/index.mjs",
    "require": "./dist/index.cjs"
  },
  "type": "module"
}
```

**Importing CommonJS from ESM:**
```javascript
// ESM file
import cjsModule from 'cjs-package'; // Works via default import
import { named } from 'cjs-package';  // May fail if not statically analyzable

// Use createRequire for full CommonJS compatibility
import { createRequire } from 'node:module';
const require = createRequire(import.meta.url);
const pkg = require('cjs-package');
```

**Importing ESM from CommonJS:**
```javascript
// CommonJS file
// Cannot use require() for ESM; must use dynamic import()
const esmModule = await import('esm-package');
```

### `__dirname` and `__filename` in ESM

```javascript
// ESM equivalent
import { fileURLToPath } from 'node:url';
import { dirname } from 'node:path';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

## 2. Loader Hooks

Loader hooks allow you to **customize the module loading process**. They can intercept and transform source code, redirect imports, or provide virtual modules.

### Custom Loader Example (Node.js 20+)

Loaders are registered via the `--import` flag or programmatically.

```javascript
// loader.mjs
import { readFile } from 'node:fs/promises';
import { fileURLToPath } from 'node:url';

export async function resolve(specifier, context, nextResolve) {
  console.log(`Resolving: ${specifier}`);
  
  // Redirect imports starting with '@virtual/' to an internal module
  if (specifier.startsWith('@virtual/')) {
    return {
      shortCircuit: true,
      url: `virtual://${specifier}`
    };
  }
  
  // Defer to default resolution
  return nextResolve(specifier, context);
}

export async function load(url, context, nextLoad) {
  console.log(`Loading: ${url}`);
  
  // Handle virtual modules
  if (url.startsWith('virtual://')) {
    const virtualName = url.replace('virtual://', '');
    return {
      shortCircuit: true,
      format: 'module',
      source: `export default { name: '${virtualName}', timestamp: ${Date.now()} };`
    };
  }
  
  // Transform TypeScript on the fly (simplified example)
  if (url.endsWith('.ts')) {
    const path = fileURLToPath(url);
    const source = await readFile(path, 'utf8');
    const transformed = source
      .replace(/: string/g, '')
      .replace(/interface \w+ \{[\s\S]*?\}/g, '');
    
    return {
      shortCircuit: true,
      format: 'module',
      source: transformed
    };
  }
  
  return nextLoad(url, context);
}
```

**Usage:**
```bash
node --import ./loader.mjs app.js
```

### Loader Hooks Lifecycle

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Module Loading Process                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. resolve(specifier, context, nextResolve)                                │
│     └─ Determine final URL of the module                                    │
│                                                                              │
│  2. load(url, context, nextLoad)                                            │
│     └─ Fetch and return source code                                         │
│                                                                              │
│  3. (Internal) Parse source based on format                                 │
│                                                                              │
│  4. (Internal) Instantiate — create module record                           │
│                                                                              │
│  5. (Internal) Evaluate — execute module code                               │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Advanced: HTTP Imports with Loader Hooks

```javascript
// http-loader.mjs
export async function resolve(specifier, context, nextResolve) {
  if (specifier.startsWith('https://')) {
    return {
      shortCircuit: true,
      url: specifier
    };
  }
  return nextResolve(specifier, context);
}

export async function load(url, context, nextLoad) {
  if (url.startsWith('https://')) {
    const response = await fetch(url);
    const source = await response.text();
    return {
      shortCircuit: true,
      format: 'module',
      source
    };
  }
  return nextLoad(url, context);
}
```

### Using `module.register()` (Node.js 20+)

```javascript
// register.js
import { register } from 'node:module';

register('./loader.mjs', import.meta.url);
```

## 3. Circular Dependencies

Circular dependencies occur when module A imports module B, and module B imports module A (directly or indirectly).

### How CommonJS Handles Circular Dependencies

CommonJS **partially** supports circular dependencies by returning the **incomplete exports object** when a module is required while it's still being executed.

```javascript
// a.js
console.log('a starting');
exports.done = false;
const b = require('./b.js');
console.log('in a, b.done =', b.done);
exports.done = true;
console.log('a done');

// b.js
console.log('b starting');
exports.done = false;
const a = require('./a.js');
console.log('in b, a.done =', a.done);
exports.done = true;
console.log('b done');

// main.js
console.log('main starting');
const a = require('./a.js');
const b = require('./b.js');
console.log('in main, a.done =', a.done, 'b.done =', b.done);
```

**Output:**
```
main starting
a starting
b starting
in b, a.done = false
b done
in a, b.done = true
a done
in main, a.done = true b.done = true
```

**What happens:**
1. `main.js` requires `a.js`
2. `a.js` starts executing, sets `exports.done = false`
3. `a.js` requires `b.js`
4. `b.js` starts executing, sets `exports.done = false`
5. `b.js` requires `a.js` — Node.js detects circular dependency and returns the **current `a.js` exports object** (with `done = false`)
6. `b.js` finishes, setting `done = true`
7. `a.js` resumes, sees `b.done = true`, sets its own `done = true`

**Pitfall:** If you replace `module.exports` (instead of mutating it), the circular reference breaks:

```javascript
// ❌ Breaks circular dependency
module.exports = class MyClass { ... };

// ✅ Works with circular dependency
module.exports.MyClass = class MyClass { ... };
```

### How ESM Handles Circular Dependencies

ESM uses **live bindings**, which makes circular dependencies more robust. When a value is imported, it's a **live reference** to the exported binding.

```javascript
// a.mjs
import { bDone, setBDone } from './b.mjs';

console.log('a starting');
export let aDone = false;

setBDone(true);  // Modifies bDone in b.mjs

console.log('in a, bDone =', bDone);
aDone = true;
console.log('a done');

// b.mjs
import { aDone } from './a.mjs';

console.log('b starting');
export let bDone = false;

export function setBDone(value) {
  bDone = value;
}

console.log('in b, aDone =', aDone);
console.log('b done');
```

**Output:**
```
b starting
in b, aDone = undefined
b done
a starting
in a, bDone = false
a done
```

**Key difference:** ESM hoists imports and establishes bindings before execution. This allows circular dependencies as long as you don't **use** the imported value before it's initialized (Temporal Dead Zone).

### Best Practices for Circular Dependencies

1. **Avoid them entirely** — Refactor shared code into a third module
2. **Use dependency inversion** — Depend on abstractions, not concrete modules
3. **If unavoidable, mutate exports rather than reassign** (CommonJS)
4. **In ESM, separate side effects from exports** to avoid TDZ issues

**Refactoring Example:**
```javascript
// Before: Circular
// a.js → requires b.js → requires a.js

// After: Extract shared code
// shared.js
export const config = {};

// a.js
import { config } from './shared.js';

// b.js
import { config } from './shared.js';
```

## 4. Module Caching

Node.js **caches modules** after the first `require()` or `import`. Subsequent imports return the same module instance.

### How Module Caching Works (CommonJS)

```javascript
// counter.js
let count = 0;
module.exports = {
  increment: () => ++count,
  getCount: () => count
};

// main.js
const counter1 = require('./counter');
const counter2 = require('./counter');

console.log(counter1 === counter2); // true — same object

counter1.increment();
console.log(counter2.getCount()); // 1 — shared state
```

**Cache location:** `require.cache`

```javascript
console.log(require.cache);
// {
//   '/path/to/counter.js': Module { ... },
//   '/path/to/main.js': Module { ... }
// }
```

### Clearing the Module Cache

```javascript
// Force reload of a module
delete require.cache[require.resolve('./counter')];
const freshCounter = require('./counter');
```

**Use case:** Hot module replacement (HMR) in development.

### ESM Caching

ESM also caches modules, but the cache is **not directly accessible**. The same module is evaluated only once per realm.

```javascript
// counter.mjs
export let count = 0;
export function increment() { count++; }

// main.mjs
import { count, increment } from './counter.mjs';
import './counter.mjs' assert { type: 'javascript' }; // Same instance

increment();
console.log(count); // 1 — live binding reflects change
```

### Caching and Memory Leaks

Module caching can cause **memory leaks** if modules hold onto large data structures.

```javascript
// ❌ Leaky module — caches data that never gets cleaned up
const cache = new Map();

module.exports = {
  set(key, value) { cache.set(key, value); },
  get(key) { return cache.get(key); }
};

// ✅ Better: Use WeakMap or provide a cleanup method
const cache = new WeakMap(); // Automatically GC'd when keys are unreachable
```

### Deterministic vs Non‑Deterministic Modules

```javascript
// ❌ Non‑deterministic — exports a new instance each time? No, cached!
module.exports = new (class Database {
  constructor() { this.connected = false; }
})();

// All imports share the same instance — might be intended (singleton)

// ✅ Explicit singleton pattern
let instance = null;
class Database {
  constructor() {
    if (instance) return instance;
    instance = this;
    this.connected = false;
  }
}
module.exports = Database;
```

## 5. Dependency Injection (DI)

Dependency Injection is a pattern where a module receives its dependencies from outside rather than creating them internally. This improves testability, modularity, and flexibility.

### Without Dependency Injection (Tightly Coupled)

```javascript
// user-service.js
const db = require('./database'); // Hard dependency
const emailService = require('./email');

class UserService {
  async createUser(userData) {
    const user = await db.users.insert(userData);
    await emailService.sendWelcome(user.email);
    return user;
  }
}
module.exports = new UserService();
```

**Problems:**
- Cannot test with a mock database
- Cannot swap database implementation
- Hard to configure differently per environment

### With Dependency Injection

```javascript
// user-service.js
class UserService {
  constructor({ db, emailService }) {
    this.db = db;
    this.emailService = emailService;
  }
  
  async createUser(userData) {
    const user = await this.db.users.insert(userData);
    await this.emailService.sendWelcome(user.email);
    return user;
  }
}

module.exports = UserService;
```

**Composition root (where dependencies are wired):**
```javascript
// container.js
const Database = require('./database');
const EmailService = require('./email');
const UserService = require('./user-service');

const db = new Database(process.env.DATABASE_URL);
const emailService = new EmailService(process.env.SENDGRID_API_KEY);
const userService = new UserService({ db, emailService });

module.exports = { userService, db, emailService };
```

### DI Container (IoC Container)

A lightweight DI container manages dependency resolution and lifecycle.

```javascript
// container.js
class Container {
  constructor() {
    this.registrations = new Map();
    this.instances = new Map();
  }
  
  register(name, factory, dependencies = [], lifecycle = 'transient') {
    this.registrations.set(name, { factory, dependencies, lifecycle });
  }
  
  registerValue(name, value) {
    this.instances.set(name, value);
  }
  
  resolve(name) {
    // Check for registered value
    if (this.instances.has(name)) {
      return this.instances.get(name);
    }
    
    const registration = this.registrations.get(name);
    if (!registration) {
      throw new Error(`No registration for: ${name}`);
    }
    
    const { factory, dependencies, lifecycle } = registration;
    
    // Resolve dependencies recursively
    const resolvedDeps = dependencies.map(dep => this.resolve(dep));
    
    const instance = factory(...resolvedDeps);
    
    if (lifecycle === 'singleton') {
      this.instances.set(name, instance);
    }
    
    return instance;
  }
  
  createScope() {
    return new Container(); // New container with separate instance cache
  }
}

// Usage
const container = new Container();

container.registerValue('config', {
  databaseUrl: process.env.DATABASE_URL,
  sendgridKey: process.env.SENDGRID_API_KEY
});

container.register('db', (config) => new Database(config.databaseUrl), 
  ['config'], 'singleton');

container.register('emailService', (config) => new EmailService(config.sendgridKey),
  ['config'], 'singleton');

container.register('userService', (db, emailService) => new UserService({ db, emailService }),
  ['db', 'emailService'], 'singleton');

// In your app
const userService = container.resolve('userService');
```

### Awilix — Production‑Ready DI for Node.js

```javascript
const awilix = require('awilix');
const { createContainer, asClass, asFunction, asValue, Lifetime } = awilix;

const container = createContainer();

container.register({
  config: asValue({
    databaseUrl: process.env.DATABASE_URL,
    sendgridKey: process.env.SENDGRID_API_KEY
  }),
  
  db: asClass(Database)
    .inject(() => ({ connectionString: container.resolve('config').databaseUrl }))
    .singleton(),
  
  emailService: asClass(EmailService)
    .inject(() => ({ apiKey: container.resolve('config').sendgridKey }))
    .singleton(),
  
  userService: asClass(UserService).singleton()
});

// Auto‑wiring by parameter name
container.register({
  userService: asClass(UserService)
    .inject(() => ({})) // Dependencies matched by constructor parameter names
    .singleton()
});

const userService = container.resolve('userService');
```

### Testing with Dependency Injection

```javascript
// user-service.test.js
const UserService = require('./user-service');

describe('UserService', () => {
  it('creates user and sends welcome email', async () => {
    // Mock dependencies
    const mockDb = {
      users: { insert: jest.fn().mockResolvedValue({ id: 1, email: 'test@example.com' }) }
    };
    const mockEmailService = {
      sendWelcome: jest.fn().mockResolvedValue(true)
    };
    
    const service = new UserService({ db: mockDb, emailService: mockEmailService });
    
    const user = await service.createUser({ email: 'test@example.com' });
    
    expect(user.id).toBe(1);
    expect(mockDb.users.insert).toHaveBeenCalled();
    expect(mockEmailService.sendWelcome).toHaveBeenCalledWith('test@example.com');
  });
});
```

### Service Locator Pattern (Alternative to DI)

```javascript
// service-locator.js
class ServiceLocator {
  constructor() {
    this.services = new Map();
  }
  
  register(name, instance) {
    this.services.set(name, instance);
  }
  
  get(name) {
    const service = this.services.get(name);
    if (!service) throw new Error(`Service not found: ${name}`);
    return service;
  }
}

const locator = new ServiceLocator();
locator.register('db', new Database());
locator.register('email', new EmailService());

// In a module
const db = require('./service-locator').get('db');
```

**Trade‑offs:**
- DI: Dependencies explicit in constructor, easier to test, more boilerplate
- Service Locator: Less boilerplate, implicit dependencies, harder to test

## 6. Module Resolution Algorithm

Understanding how Node.js finds modules helps debug `MODULE_NOT_FOUND` errors.

### Resolution Steps for `require('xyz')`

1. **Core modules**: Check if `xyz` is a built‑in module (`fs`, `http`, etc.)
2. **Relative/Absolute paths**: `./xyz`, `/xyz`, `../xyz`
   - Try exact file
   - Try adding `.js`, `.json`, `.node` extensions
   - Try as directory with `package.json` `main` or `index.js`
3. **Node modules**: Walk up directory tree looking for `node_modules/xyz`
4. **Global folders**: `$NODE_PATH` or global install locations

### ESM Resolution Differences

- Must use full file extensions (`.js`, `.mjs`) unless using an import map or loader
- `index.js` is not automatically resolved
- Directory imports require `package.json` with `"type": "module"` and `"exports"`

### Customizing Resolution with `exports` Field

```json
{
  "name": "my-package",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./feature": {
      "import": "./dist/feature.mjs",
      "require": "./dist/feature.cjs"
    },
    "./package.json": "./package.json"
  }
}
```

This **encapsulates** the package — consumers cannot import internal files not listed in `exports`.

## Summary Table

| Concept | Key Insight |
|---------|-------------|
| **CommonJS** | Synchronous, `require()` caches, mutable exports object |
| **ESM** | Asynchronous, static analysis, live bindings, tree‑shakable |
| **Loader Hooks** | Intercept module resolution and loading for custom behavior |
| **Circular Dependencies** | Avoid when possible; CommonJS returns partial exports, ESM uses TDZ |
| **Module Caching** | Modules execute once; clear cache for hot reload; beware memory leaks |
| **Dependency Injection** | Improves testability and flexibility; use Awilix or manual composition |
| **Module Resolution** | Understand algorithm to debug missing modules; use `exports` field for encapsulation |

**Key Takeaways:**
1. **Prefer ESM for new projects** — It's the standard and enables tree‑shaking
2. **Use `exports` field** to control your package's public API
3. **Avoid circular dependencies** — Refactor to a shared module
4. **Leverage module caching** for singletons, but provide cleanup when needed
5. **Adopt dependency injection** for testable, modular architecture
6. **Use loader hooks sparingly** — They're powerful but can complicate debugging
