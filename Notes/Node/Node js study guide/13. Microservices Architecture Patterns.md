# Microservices Architecture Patterns – Node.js Guide

Microservices architecture decomposes an application into small, independent services that communicate over the network. This approach enables scalability, fault isolation, and team autonomy but introduces complexity in coordination, data consistency, and observability. This guide covers the essential patterns for building production‑grade microservices with Node.js.

---

## 1. Service Discovery

In a microservices environment, services are **dynamic**: instances come and go due to scaling, failures, or deployments. Hard‑coding IP addresses is impossible. Service discovery solves this by providing a **registry** where services register themselves and clients look up available instances.

### Client‑Side Discovery

The client queries a **service registry** to get the location of a service instance, then makes the request directly.

```
┌─────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Client    │────▶│ Service Registry│     │  Service        │
│ (API Gateway│     │  (Consul/Etcd)  │     │  Instance A     │
│   or other  │     └────────┬────────┘     └────────┬────────┘
│   service)  │              │                       │
└─────────────┘              │ lookup("users")       │
       │                     └───────────────────────┘
       │                              │
       │       direct request         │
       └──────────────────────────────┘
```

**Implementation with Consul and Node.js:**

```javascript
const Consul = require('consul');
const express = require('express');

const consul = new Consul({ host: 'consul-server', port: 8500 });
const app = express();

const serviceId = `user-service-${process.pid}`;
const port = process.env.PORT || 3000;

// Register service on startup
app.listen(port, async () => {
  await consul.agent.service.register({
    name: 'user-service',
    id: serviceId,
    address: 'user-service-host',
    port: port,
    check: {
      http: `http://user-service-host:${port}/health`,
      interval: '10s',
      timeout: '5s',
      deregistercriticalserviceafter: '30s'
    }
  });
  console.log(`User service registered on port ${port}`);
});

// Health check endpoint
app.get('/health', (req, res) => res.send('OK'));

// Deregister on shutdown
process.on('SIGTERM', async () => {
  await consul.agent.service.deregister(serviceId);
  process.exit(0);
});

// Client lookup
async function getUserServiceUrl() {
  const services = await consul.catalog.service.nodes('user-service');
  const healthyNodes = services.filter(node =>
    node.Checks.every(check => check.Status === 'passing')
  );
  if (healthyNodes.length === 0) throw new Error('No healthy user service');
  const node = healthyNodes[Math.floor(Math.random() * healthyNodes.length)];
  return `http://${node.ServiceAddress}:${node.ServicePort}`;
}
```

### Server‑Side Discovery

The client makes a request to a **load balancer** (e.g., Envoy, Nginx, Kubernetes Service), which queries the registry and routes the request.

```
┌─────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Client    │────▶│  Load Balancer  │────▶│  Service        │
│             │     │  (Nginx/Envoy)  │     │  Instance A     │
└─────────────┘     └────────┬────────┘     └─────────────────┘
                             │
                             │ lookup from registry
                             ▼
                    ┌─────────────────┐
                    │ Service Registry│
                    └─────────────────┘
```

### Service Discovery in Kubernetes

Kubernetes provides built‑in service discovery via **DNS** and **environment variables**.

```javascript
// In Kubernetes, simply use the service name
const USER_SERVICE_URL = process.env.USER_SERVICE_URL || 'http://user-service:3000';

// Kubernetes DNS resolves "user-service" to the ClusterIP
const response = await fetch(`${USER_SERVICE_URL}/users`);
```

### Service Mesh (Istio / Linkerd)

A service mesh offloads service discovery, load balancing, retries, and observability from the application code to a sidecar proxy.

```yaml
# No code changes needed — the sidecar handles it
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service
  ports:
    - port: 3000
```

## 2. Event Sourcing

Event sourcing persists the **state changes** (events) rather than the current state. The current state is reconstructed by replaying all events. This provides a **complete audit trail** and enables temporal queries.

### Core Concepts

- **Event**: Immutable record of something that happened (e.g., `UserCreated`, `EmailChanged`)
- **Event Store**: Append‑only log of all events
- **Projection**: Derived view of current state by applying events
- **Replay**: Rebuilding state from the event stream

```
┌─────────────┐     ┌─────────────────────────────────────┐
│   Command   │────▶│          Event Store                │
│ "UpdateEmail│     │ [UserCreated, EmailChanged, ...]    │
└─────────────┘     └───────────────┬─────────────────────┘
                                    │
                                    │ subscribe
                                    ▼
                          ┌─────────────────┐
                          │   Projections   │
                          │ (Current State) │
                          └─────────────────┘
```

### Implementation with Node.js and EventStoreDB

```javascript
const { EventStoreDBClient, StreamNotFoundError } = require('@eventstore/db-client');

const client = EventStoreDBClient.connectionString('esdb://localhost:2113?tls=false');

// Append events
async function appendUserEvents(userId, events, expectedVersion) {
  const streamName = `user-${userId}`;

  const eventData = events.map(event => ({
    type: event.type,
    data: event.data
  }));

  return client.appendToStream(streamName, eventData, {
    expectedRevision: expectedVersion // Optimistic concurrency
  });
}

// Read events and build projection
async function getUserProjection(userId) {
  const streamName = `user-${userId}`;
  const events = client.readStream(streamName);

  const user = { id: userId, email: null, name: null, version: 0 };

  for await (const { event } of events) {
    user.version++;
    switch (event.type) {
      case 'UserCreated':
        user.email = event.data.email;
        user.name = event.data.name;
        break;
      case 'EmailChanged':
        user.email = event.data.newEmail;
        break;
      case 'NameChanged':
        user.name = event.data.newName;
        break;
    }
  }

  return user;
}

// Command handler
async function changeEmail(userId, newEmail) {
  const current = await getUserProjection(userId);

  const event = {
    type: 'EmailChanged',
    data: { userId, newEmail, changedAt: new Date() }
  };

  // Append with expected version (optimistic concurrency)
  await appendUserEvents(userId, [event], current.version);
}
```

### Building a Projection (Read Model) with Kafka

Events can be published to Kafka for other services to consume and build their own projections.

```javascript
const { Kafka } = require('kafkajs');

const kafka = new Kafka({ clientId: 'user-service', brokers: ['localhost:9092'] });
const producer = kafka.producer();

async function appendUserEvents(userId, events, expectedVersion) {
  // 1. Append to EventStoreDB
  await eventStore.append(...);

  // 2. Publish to Kafka for projections
  for (const event of events) {
    await producer.send({
      topic: 'user-events',
      messages: [{
        key: userId,
        value: JSON.stringify(event),
        headers: { 'event-type': event.type }
      }]
    });
  }
}

// Consumer building a materialized view
const consumer = kafka.consumer({ groupId: 'user-projection' });

await consumer.subscribe({ topic: 'user-events', fromBeginning: true });

const userView = new Map(); // In production, use a database

await consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value.toString());
    const userId = message.key.toString();

    switch (message.headers['event-type']) {
      case 'UserCreated':
        userView.set(userId, { id: userId, ...event.data });
        break;
      case 'EmailChanged':
        const user = userView.get(userId);
        if (user) user.email = event.data.newEmail;
        break;
    }
  }
});
```

### Advantages and Challenges

| Advantage | Challenge |
|-----------|-----------|
| Complete audit trail | Eventual consistency of projections |
| Temporal queries (state at any time) | Event schema evolution |
| Easy to add new projections later | Increased storage (all events kept) |
| Natural fit with CQRS | Learning curve |

## 3. CQRS (Command Query Responsibility Segregation)

CQRS separates **write operations** (commands that change state) from **read operations** (queries that return data). This allows optimizing each side independently.

### Architecture Overview

```
┌─────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Command   │────▶│  Command Model  │────▶│   Event Store   │
│   (Write)   │     │  (Domain Logic) │     │                 │
└─────────────┘     └─────────────────┘     └────────┬────────┘
                                                      │ events
                                                      ▼
┌─────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Query     │────▶│   Read Model    │◀────│   Projection    │
│   (Read)    │     │  (Optimized DB) │     │   (Materialized │
└─────────────┘     └─────────────────┘     │      View)      │
                                            └─────────────────┘
```

### Node.js Implementation with Separate Databases

```javascript
// Command side — uses EventStoreDB (append‑only)
class UserCommandService {
  async createUser(command) {
    const event = { type: 'UserCreated', data: command };
    await eventStore.append(`user-${command.userId}`, event);
  }

  async updateEmail(command) {
    const current = await this.getCurrentState(command.userId);
    if (current.email === command.newEmail) return;

    const event = { type: 'EmailChanged', data: { newEmail: command.newEmail } };
    await eventStore.append(`user-${command.userId}`, event);
  }
}

// Query side — uses MongoDB (optimized for reads)
class UserQueryService {
  constructor() {
    this.db = mongoose.connection;
    this.subscribeToEvents();
  }

  subscribeToEvents() {
    const consumer = kafka.consumer({ groupId: 'user-query-projection' });
    consumer.run({
      eachMessage: async ({ message }) => {
        const event = JSON.parse(message.value);
        await this.applyEvent(event);
      }
    });
  }

  async applyEvent(event) {
    switch (event.type) {
      case 'UserCreated':
        await this.db.collection('users').insertOne(event.data);
        break;
      case 'EmailChanged':
        await this.db.collection('users').updateOne(
          { userId: event.data.userId },
          { $set: { email: event.data.newEmail } }
        );
        break;
    }
  }

  async getUserById(userId) {
    return this.db.collection('users').findOne({ userId });
  }

  async searchUsers(filters) {
    return this.db.collection('users').find(filters).toArray();
  }
}
```

### Benefits of CQRS

- **Independent scaling**: Scale read side heavily for high query volume
- **Optimized data models**: Reads can use denormalized, query‑friendly schemas
- **Separation of concerns**: Commands enforce business rules; queries are simple
- **Improved security**: Commands can have stricter authorization

### When NOT to Use CQRS

- Simple CRUD applications
- Tight consistency requirements (eventual consistency may be unacceptable)
- Small teams where complexity outweighs benefits

## 4. Saga Patterns

A **saga** is a sequence of local transactions that together fulfill a business process spanning multiple services. Each step has a **compensating action** to undo changes if something fails.

### Two Implementation Styles

| Style | Coordination | Example |
|-------|--------------|---------|
| **Choreography** | Services react to events; no central coordinator | Each service listens and performs its step |
| **Orchestration** | Central orchestrator tells each service what to do | Saga execution coordinator |

### Choreography Example (Order Processing)

```javascript
// Order Service — initiates saga
async function createOrder(orderData) {
  const order = { id: uuid(), ...orderData, status: 'pending' };
  await db.orders.insert(order);

  // Publish event
  await kafka.send({
    topic: 'order-events',
    messages: [{ key: order.id, value: JSON.stringify({
      type: 'OrderCreated',
      data: order
    })}]
  });
}

// Payment Service — reacts to OrderCreated
consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value);
    if (event.type === 'OrderCreated') {
      try {
        await processPayment(event.data);
        await kafka.send({
          topic: 'payment-events',
          messages: [{ key: event.data.id, value: JSON.stringify({
            type: 'PaymentSucceeded',
            data: { orderId: event.data.id }
          })}]
        });
      } catch (err) {
        await kafka.send({
          topic: 'payment-events',
          messages: [{ key: event.data.id, value: JSON.stringify({
            type: 'PaymentFailed',
            data: { orderId: event.data.id, reason: err.message }
          })}]
        });
      }
    }
  }
});

// Order Service — listens for payment outcome
consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value);
    if (event.type === 'PaymentSucceeded') {
      await db.orders.updateOne(
        { id: event.data.orderId },
        { $set: { status: 'confirmed' } }
      );
    } else if (event.type === 'PaymentFailed') {
      // Compensating action: cancel order
      await db.orders.updateOne(
        { id: event.data.orderId },
        { $set: { status: 'cancelled' } }
      );
      // Publish compensation event for other services
      await kafka.send({ topic: 'order-events', messages: [{
        value: JSON.stringify({ type: 'OrderCancelled', data: { orderId: event.data.orderId } })
      }]});
    }
  }
});
```

### Orchestration Example with a Saga Orchestrator

```javascript
class OrderSagaOrchestrator {
  constructor() {
    this.steps = [
      { service: 'inventory', action: 'reserve', compensate: 'release' },
      { service: 'payment', action: 'charge', compensate: 'refund' },
      { service: 'shipping', action: 'create', compensate: 'cancel' }
    ];
    this.executions = new Map();
  }

  async start(orderId, orderData) {
    const execution = {
      orderId,
      data: orderData,
      currentStep: 0,
      completedSteps: [],
      status: 'running'
    };
    this.executions.set(orderId, execution);

    return this.next(execution);
  }

  async next(execution) {
    if (execution.currentStep >= this.steps.length) {
      execution.status = 'completed';
      return { status: 'completed' };
    }

    const step = this.steps[execution.currentStep];
    try {
      const result = await this.callService(step.service, step.action, execution.data);
      execution.completedSteps.push({ ...step, result });
      execution.currentStep++;
      return this.next(execution);
    } catch (err) {
      return this.compensate(execution, err);
    }
  }

  async compensate(execution, error) {
    console.error(`Saga failed at step ${execution.currentStep}:`, error);

    // Run compensations in reverse order
    for (let i = execution.completedSteps.length - 1; i >= 0; i--) {
      const step = execution.completedSteps[i];
      try {
        await this.callService(step.service, step.compensate, execution.data);
      } catch (compErr) {
        console.error(`Compensation failed for ${step.service}:`, compErr);
        // Log to dead letter queue for manual intervention
      }
    }

    execution.status = 'failed';
    return { status: 'failed', error };
  }

  async callService(service, action, data) {
    // In production, use service discovery and HTTP/gRPC
    const url = `http://${service}-service/${action}`;
    const response = await fetch(url, {
      method: 'POST',
      body: JSON.stringify(data),
      headers: { 'Content-Type': 'application/json' }
    });
    if (!response.ok) throw new Error(`${service}.${action} failed`);
    return response.json();
  }
}
```

### Handling Failures and Idempotency

Saga participants must be **idempotent** to handle duplicate messages or retries.

```javascript
// Idempotent consumer using processed message IDs
const processedMessages = new Set(); // In production: Redis with TTL

consumer.run({
  eachMessage: async ({ topic, partition, message }) => {
    const messageId = message.headers['message-id'];
    if (processedMessages.has(messageId)) {
      console.log('Duplicate message, skipping');
      return;
    }

    try {
      await processEvent(message);
      processedMessages.add(messageId);
      // Store in Redis: await redis.setex(`processed:${messageId}`, 3600, '1');
    } catch (err) {
      // Will retry; idempotency prevents double‑processing
    }
  }
});
```

## 5. Message Queues in Microservices

Message queues are the backbone of asynchronous communication in microservices. They enable **loose coupling**, **reliability**, and **scalability**.

### Common Patterns

#### Request‑Reply (Synchronous over Async)

```javascript
// Using NATS for request‑reply
const { connect } = require('nats');

const nc = await connect({ servers: ['nats://localhost:4222'] });

// Service
nc.subscribe('users.get', {
  callback: (err, msg) => {
    const userId = msg.string();
    const user = db.users.get(userId);
    msg.respond(JSON.stringify(user));
  }
});

// Client
const reply = await nc.request('users.get', 'user123', { timeout: 1000 });
const user = JSON.parse(reply.string());
```

#### Competing Consumers (Work Queue)

Multiple instances of a service consume from the same queue; each message is processed exactly once (at‑least‑once).

```javascript
// All worker instances share the same consumer group
const consumer = kafka.consumer({ groupId: 'email-sender' });
await consumer.subscribe({ topic: 'send-email' });

await consumer.run({
  eachMessage: async ({ message }) => {
    await sendEmail(JSON.parse(message.value));
    // Offset committed automatically or manually
  }
});
```

#### Publish‑Subscribe (Event Broadcasting)

```javascript
// Producer
await producer.send({
  topic: 'user-events',
  messages: [{ value: JSON.stringify(event) }]
});

// Multiple consumers with different groupIds each get a copy
const auditConsumer = kafka.consumer({ groupId: 'audit-logger' });
const notificationConsumer = kafka.consumer({ groupId: 'notification-service' });
```

#### Dead Letter Queue (DLQ)

```javascript
// After max retries, send to DLQ
const maxRetries = 3;
const retryCount = message.headers['retry-count'] || 0;

try {
  await processMessage(message);
} catch (err) {
  if (retryCount < maxRetries) {
    await producer.send({
      topic: 'order-events',
      messages: [{
        value: message.value,
        headers: { 'retry-count': (retryCount + 1).toString() }
      }]
    });
  } else {
    await producer.send({
      topic: 'order-events.dlq',
      messages: [{
        value: message.value,
        headers: { 'error': err.message }
      }]
    });
  }
}
```

### Choosing a Message Broker

| Broker | Best For | Node.js Library |
|--------|----------|-----------------|
| **RabbitMQ** | Complex routing, exactly‑once delivery | `amqplib` |
| **Kafka** | Event streaming, high throughput, replay | `kafkajs` |
| **NATS** | Low latency, simplicity, request‑reply | `nats` |
| **Redis Pub/Sub** | Lightweight, ephemeral messaging | `ioredis` |
| **AWS SQS/SNS** | Managed, integration with AWS | `@aws-sdk/client-sqs` |

## 6. Observability in Microservices

Observability answers **what's happening inside your distributed system**. It comprises three pillars: **logs**, **metrics**, and **traces**.

### Structured Logging

```javascript
const winston = require('winston');

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  defaultMeta: { service: 'user-service' },
  transports: [new winston.transports.Console()]
});

// Include trace ID in every log (from request context)
app.use((req, res, next) => {
  const traceId = req.headers['x-trace-id'] || uuid();
  req.traceId = traceId;
  logger.defaultMeta.traceId = traceId;
  next();
});

logger.info('User created', { userId: '123', email: 'user@example.com' });
```

### Metrics with Prometheus

```javascript
const client = require('prom-client');

// Create metrics
const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Duration of HTTP requests in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.1, 0.5, 1, 2, 5]
});

const activeConnections = new client.Gauge({
  name: 'active_connections',
  help: 'Number of active connections'
});

// Expose metrics endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', client.register.contentType);
  res.end(await client.register.metrics());
});

// Middleware to record metrics
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer();
  res.on('finish', () => {
    end({ method: req.method, route: req.route?.path, status_code: res.statusCode });
  });
  next();
});
```

### Distributed Tracing with OpenTelemetry

```javascript
const { NodeTracerProvider } = require('@opentelemetry/sdk-trace-node');
const { SimpleSpanProcessor } = require('@opentelemetry/sdk-trace-base');
const { JaegerExporter } = require('@opentelemetry/exporter-jaeger');
const { trace, context, propagation } = require('@opentelemetry/api');

// Setup tracer
const provider = new NodeTracerProvider();
provider.addSpanProcessor(new SimpleSpanProcessor(new JaegerExporter({
  endpoint: 'http://jaeger:14268/api/traces'
})));
provider.register();

const tracer = trace.getTracer('user-service');

// Middleware to extract trace context from incoming requests
app.use((req, res, next) => {
  const extractedContext = propagation.extract(context.active(), req.headers);
  context.with(extractedContext, () => next());
});

// Instrument a request
app.get('/users/:id', async (req, res) => {
  const span = tracer.startSpan('getUserById');
  span.setAttribute('user.id', req.params.id);

  try {
    const user = await db.users.findOne({ id: req.params.id });
    span.setAttribute('user.found', !!user);
    res.json(user);
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: SpanStatusCode.ERROR });
    throw err;
  } finally {
    span.end();
  }
});

// Propagate trace context to downstream services
async function callOrderService(userId) {
  const span = tracer.startSpan('callOrderService');
  const headers = {};
  propagation.inject(trace.setSpan(context.active(), span), headers);

  const response = await fetch('http://order-service/orders', { headers });
  span.end();
  return response.json();
}
```

### Circuit Breaker Pattern

Protect services from cascading failures using a circuit breaker.

```javascript
const CircuitBreaker = require('opossum');

const breaker = new CircuitBreaker(async (userId) => {
  const response = await fetch(`http://user-service/users/${userId}`);
  return response.json();
}, {
  timeout: 3000,              // Max request time
  errorThresholdPercentage: 50, // Open after 50% failures
  resetTimeout: 30000,        // Try again after 30s
  rollingCountTimeout: 10000, // 10‑second window
  rollingCountBuckets: 10
});

breaker.fallback(() => ({ id: 'unknown', name: 'Service unavailable' }));

breaker.on('open', () => logger.warn('Circuit breaker opened'));
breaker.on('close', () => logger.info('Circuit breaker closed'));
breaker.on('halfOpen', () => logger.info('Circuit breaker half‑open'));

app.get('/users/:id', async (req, res) => {
  const user = await breaker.fire(req.params.id);
  res.json(user);
});
```

**Key Takeaways:**
1. **Embrace eventual consistency** — distributed transactions are hard; use sagas
2. **Design for failure** — circuit breakers, retries, idempotency
3. **Observability is not optional** — trace every request, log with context
4. **Choose the right message broker** — Kafka for events, RabbitMQ for reliability, NATS for speed
5. **Service discovery is handled by the platform** — use Kubernetes DNS in production
6. **CQRS + Event Sourcing are powerful but complex** — use only when justified
